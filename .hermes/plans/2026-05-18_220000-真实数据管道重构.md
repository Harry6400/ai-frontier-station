# 真实数据管道重构计划

## 问题

当前种子数据是我编造的虚假内容，不是真实数据。用户需要：
1. 从真实来源（arXiv/GitHub/Arena/RSS等）获取真实数据
2. 数据来源和处理过程可编辑（不只是最终文章）
3. 产品动态按产品归属组织

## 现有真实数据管道（仅论文）

```
arXiv API → ArxivService → 保存原始候选 → PaperSummaryService(MiMo) → 生成中文摘要 → Content表
HuggingFace API → HuggingFaceService → 保存原始候选 → 同上
```

后台已有"arXiv论文导入"和"HuggingFace导入"对话框，能搜索候选、预览、确认导入。**论文管道是真实的**。

非论文板块完全没有管道，种子数据是假的。

## 目标架构

```
数据源 → 原始采集(候选表) → 管理员审核/编辑 → AI处理(可选) → 最终内容(Content表)
```

关键设计：**两层数据**
- **原始层**：从API/网页直接抓取的原始数据，存在候选表中，管理员可编辑
- **处理层**：经过AI总结/格式化后的最终内容，管理员可再编辑

管理员可以看到：原始数据是什么 → AI怎么处理的 → 最终展示什么。每一步都可编辑。

## 执行步骤

### 步骤0：清理假数据
- 删除所有非论文的种子数据
- `DELETE FROM ai_content WHERE content_type != 'paper'`

### 步骤1：新增原始数据候选表

```sql
CREATE TABLE ai_content_candidate (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  source_type VARCHAR(50) NOT NULL COMMENT '来源: github/news/arena/tools/product',
  external_id VARCHAR(200) COMMENT '外部ID(如github repo full_name)',
  title VARCHAR(500) NOT NULL,
  url VARCHAR(1000),
  raw_content LONGTEXT COMMENT '原始内容(HTML/JSON/Markdown)',
  author VARCHAR(200),
  published_at DATETIME,
  metadata_json JSON COMMENT '原始结构化数据(stars/forks/elo等)',
  ai_summary TEXT COMMENT 'AI生成的摘要',
  ai_body LONGTEXT COMMENT 'AI生成的正文',
  status VARCHAR(20) DEFAULT 'pending' COMMENT 'pending/approved/rejected/imported',
  content_id BIGINT COMMENT '导入后的Content ID',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_candidate_source (source_type),
  KEY idx_candidate_status (status)
);
```

**文件变更**: `database/schema.sql` 加表

### 步骤2：后端采集服务（每个来源一个Service）

**2a. GitHub采集** (GitHubTrendingService)
- 数据源：GitHub REST API `/search/repositories` + `trending` 页面抓取
- 免费：有token 5000次/小时
- 采集内容：repo名、描述、star数、语言、README摘要
- 存入 ai_content_candidate (source_type='github')

**2b. AI资讯采集** (NewsRssService)
- 数据源：Reuters/TechCrunch/The Verge RSS feeds + Hacker News API
- 免费：RSS公开，HN Firebase API无限制
- 采集内容：标题、摘要、来源、发布时间、原文链接
- 存入 ai_content_candidate (source_type='news')

**2c. 产品动态采集** (ProductUpdateService)
- 数据源：OpenAI/Anthropic/Google/DeepSeek官方博客RSS + GitHub Release API
- 免费：RSS公开，GitHub API免费
- 采集内容：产品名、更新类型(model/api/ide/price)、标题、内容
- 存入 ai_content_candidate (source_type='product')

**2d. Arena数据** (ArenaService)
- 数据源：LMSYS Chatbot Arena 页面/API（如果有的话，否则抓取排行榜页面）
- 免费：公开页面
- 采集内容：模型名、ELO分数、排名变化、投票数
- 存入 ai_content_candidate (source_type='arena')

**2e. 工具实践采集** (ToolsRssService)
- 数据源：X/Twitter(受限)、Reddit API、Hacker News API、Product Hunt API
- 免费：Reddit/HN公开API，X有限
- 采集内容：作者、内容、平台、互动数据、链接
- 存入 ai_content_candidate (source_type='tools')

### 步骤3：后台管理——候选审核页面

复用现有的"arXiv论文导入"对话框模式：
1. 左侧：候选列表（原始数据），可筛选来源/状态
2. 右侧：选中候选后显示原始内容 + AI处理结果
3. 操作：批准→生成Content / 拒绝 / 编辑后再批准
4. 管理员可以修改AI生成的摘要和正文

**文件变更**:
- 新建 `frontend-admin/src/views/CandidateReviewView.vue`
- 新建 `backend/controller/admin/CandidateAdminController.java`
- 更新 `frontend-admin/src/router/index.js` 加路由

### 步骤4：产品实体化

在 extra_json 中用 `product` 字段标识产品归属，不需要新表：
```json
{
  "product": "ChatGPT",
  "company": "OpenAI",
  "updateType": "model"
}
```

候选表的 metadata_json 也包含产品信息，导入时自动填入 extra_json。

后台产品动态页面按产品分组显示（从extra_json.product读取）。

### 步骤5：定时采集调度

每个来源一个 Scheduler，复用现有 ArxivFetchScheduler 模式：
- GitHub：每周一 09:00
- News：每天 08:00, 20:00
- 产品动态：每天 09:00
- Arena：每天 10:00
- 工具实践：每天 12:00

**文件变更**: 新建 5 个 Scheduler 文件

### 步骤6：前端对接

6个板块从Content表读取已导入的内容（真实数据），点击进入详情页。
