# 信息架构重构计划：按板块特性分管道

> 核心理念：不同板块有不同的信息特性，不应该用同一套管道处理所有内容。

## 当前问题

所有内容都走同一条管道：`采集 → 候选表 → 聚类 → 事件 → 管理员审核`

但实际上：
- 论文每篇独立，不需要聚类
- GitHub每个项目独立，不需要聚类
- 工具帖子各不相同，不需要聚类
- Arena是固定格式数据，不需要聚类
- **只有AI资讯需要聚类**（多来源报道同一事件）
- **产品动态按产品分组**（不是通用聚类）

## 新架构：三种管道

```
管道A（自动发布）：论文 / GitHub / Arena / 工具实践
  采集 → AI处理 → 直接写入Content表 → 前台展示
  管理员不介入

管道B（事件审核）：AI资讯
  采集 → AI聚类成事件 → AI生成事件内容 → 管理员审核事件 → 发布

管道C（产品分组）：产品动态
  采集 → 按产品自动归类 → AI生成产品摘要 → 管理员审核 → 发布
```

## 管理员角色

| 板块 | 管理员介入 | 原因 |
|------|-----------|------|
| 论文 | ❌ 不需要 | 太多太频繁，自动发布 |
| GitHub | ❌ 不需要 | 自动发布 |
| Arena | ❌ 不需要 | 结构化数据，自动发布 |
| 工具实践 | ❌ 不需要 | 独立帖子，自动发布 |
| AI资讯 | ✅ 审核事件 | 需要判断聚类是否准确 |
| 产品动态 | ✅ 审核更新 | 需要判断是否重要 |

## 实施步骤

### 步骤1：管道A——自动发布服务

**新建** `backend/service/AutoPublishService.java` + `Impl`

职责：采集完成后，自动将候选转为Content并发布。

```java
public interface AutoPublishService {
    void publishCandidate(Long candidateId);  // 单条发布
    int publishPending(String sourceType);    // 批量发布某类型所有pending候选
}
```

实现逻辑：
1. 读取候选数据
2. 调用AI生成中文摘要和正文（复用现有callAiApi）
3. 直接写入Content表（status=PUBLISHED）
4. 更新候选状态为imported
5. 不经过事件聚类

**修改** `CandidateServiceImpl.fetchFromSource()`：
- 论文/GitHub/Arena/工具 采集后 → 调用 `autoPublishService.publishPending(sourceType)`
- 新闻/产品 采集后 → 调用 `eventService.autoCluster()`（仅对新闻）

### 步骤2：修改采集后处理逻辑

**修改** `CandidateServiceImpl.fetchFromSource()`：

```java
public int fetchFromSource(String sourceType) {
    int count;
    switch (sourceType) {
        case "github":  count = gitHubTrendingService.fetchTrending("", "daily"); break;
        case "news":    count = newsRssService.fetchNews(); break;
        case "product": count = productUpdateService.fetchProductUpdates(); break;
        case "tools":   count = toolsRssService.fetchTools(); break;
        case "arena":   count = arenaService.fetchArenaData(); break;
        default: throw BizException.badRequest("暂不支持: " + sourceType);
    }
    
    // 采集后处理：按类型走不同管道
    if (count > 0) {
        switch (sourceType) {
            case "github":
            case "tools":
            case "arena":
                // 管道A：自动发布
                autoPublishService.publishPending(sourceType);
                break;
            case "news":
                // 管道B：聚类成事件
                eventService.autoCluster();
                break;
            case "product":
                // 管道C：按产品分组
                productGroupService.autoGroup();
                break;
        }
    }
    return count;
}
```

### 步骤3：产品动态按产品分组

**新建** `backend/service/ProductGroupService.java` + `Impl`

职责：将产品动态候选按产品名分组，生成每个产品的综合摘要。

```java
public interface ProductGroupService {
    int autoGroup();  // 自动分组，返回新创建的事件数
}
```

实现逻辑：
1. 获取所有pending的产品动态候选
2. 按metadataJson.product分组（OpenAI/Anthropic/Google/DeepSeek等）
3. 对每个产品组，调用AI生成该产品的综合更新摘要
4. 创建事件（title="OpenAI本周更新"，description=AI生成的综合内容）
5. 关联候选到事件

### 步骤4：修改前端数据采集页面

**修改** `frontend-admin/src/views/CandidateReviewView.vue`

当前：显示所有候选 + 所有事件
改为：
- **只显示需要审核的内容**（新闻事件 + 产品事件）
- 移除候选列表（自动发布的不需要审核）
- 保留事件审核面板
- 触发采集按钮改为按板块触发

页面布局：
```
┌─────────────────────────────────────────────┐
│ [触发采集: AI资讯] [触发采集: 产品动态]      │
│ [触发采集: GitHub] [触发采集: 工具实践]      │
├─────────────────────────────────────────────┤

│ ┌─────────────────────────────────────────┐ │
│ │ 📰 OpenAI本周动态（3条新闻聚合）        │ │
│ │ AI生成的摘要内容...                      │ │
│ │ [批准] [拒绝]                            │ │
│ ├─────────────────────────────────────────┤ │
│ │ 📦 Anthropic产品更新（2条更新聚合）      │ │
│ │ AI生成的摘要内容...                      │ │
│ │ [批准] [拒绝]                            │ │
│ └─────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
```

### 步骤5：清理旧数据和代码

1. 清理已有的非论文候选（它们应该已经被自动发布）
2. EventServiceImpl.autoCluster() 只对新闻类型生效
