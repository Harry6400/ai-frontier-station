# 阶段 30：安全漏洞修复

## 阶段目标

- 修复代码审查中发现的严重和高危安全漏洞
- 提升系统的安全性和健壮性

## 本阶段完成情况

### 修复的问题

#### 严重问题（2 个）

1. **硬编码默认加密主密钥**
   - 问题：`API_MASTER_KEY` 有公开的默认值 `ai-frontier-station-default-key-2024`
   - 风险：攻击者可解密数据库中所有 API 密钥
   - 修复：移除默认值，添加启动时验证，未配置则生成随机密钥并警告

2. **硬编码默认 JWT 密钥**
   - 问题：`JWT_SECRET` 有公开的默认值
   - 风险：攻击者可伪造 JWT Token，冒充管理员
   - 修复：移除默认值，添加启动时验证，未配置则生成随机密钥并警告

#### 高危问题（3 个）

3. **arXiv XML 解析存在 XXE 漏洞**
   - 问题：`DocumentBuilderFactory` 未禁用外部实体
   - 风险：恶意 XML 可读取本地文件、SSRF 攻击
   - 修复：添加 XML 安全特性禁用外部实体

4. **JWT 密钥零字节填充**
   - 问题：短密钥用零字节填充到 32 字节，降低熵值
   - 修复：使用 SHA-256 哈希派生完整密钥

5. **密钥派生使用单次 SHA-256**
   - 说明：当前实现对课程项目可接受
   - 改进：添加注释说明生产环境应使用 PBKDF2/Argon2

### 验证结果

- [x] 后端编译通过
- [x] 安全特性正确应用

## 技术实现细节

### 1. 加密主密钥验证

```java
public static String validateOrGenerateMasterKey(String masterKey) {
    if (masterKey == null || masterKey.isBlank()) {
        String generated = generateRandomKey();
        log.warn("WARNING: API_MASTER_KEY 未配置，已生成随机密钥");
        return generated;
    }
    if (masterKey.length() < 16) {
        log.warn("WARNING: API_MASTER_KEY 长度不足 16 字符，安全性较低");
    }
    return masterKey;
}
```

### 2. JWT 密钥派生

```java
private SecretKey deriveSigningKey(String secret) {
    byte[] keyBytes = secret.getBytes(StandardCharsets.UTF_8);
    MessageDigest sha256 = MessageDigest.getInstance("SHA-256");
    byte[] hash = sha256.digest(keyBytes);
    return Keys.hmacShaKeyFor(hash);
}
```

## 答辩时可以怎么讲

### 安全意识

> 在代码审查中发现了几个安全问题，包括硬编码默认密钥和 XML 解析漏洞。这些问题在开发阶段可能不明显，但在生产环境中会被攻击者利用。通过这次修复，我学习到了安全编码的重要性。

### 修复措施

> 对于硬编码密钥问题，我移除了默认值，添加了启动时验证。如果环境变量未配置，系统会生成随机密钥并打印警告。对于 XXE 漏洞，我禁用了 XML 解析器的外部实体处理功能。

### 密钥管理

> 系统现在要求生产环境必须配置 API_MASTER_KEY 和 JWT_SECRET 环境变量。开发环境可以不配置，系统会自动生成临时密钥，但重启后会失效。这种设计既方便了开发，又保证了生产安全。

## 后续改进方向

1. **密钥派生算法**：生产环境使用 PBKDF2 或 Argon2 替代 SHA-256
2. **密钥轮换**：支持定期更换加密密钥
3. **密钥存储**：使用 HashiCorp Vault 等专业密钥管理服务
4. **安全审计**：定期进行安全代码审查
