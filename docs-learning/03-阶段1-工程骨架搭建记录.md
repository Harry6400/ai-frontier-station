# 阶段1：工程骨架搭建记录

## 1. 本阶段目标

这一阶段不是直接写完整业务，而是先把项目做成“真正的工程”。

要解决的问题是：

- 项目根目录怎么组织
- 前台、后台、后端怎么拆
- 数据库核心表怎么先定下来
- 后续怎么避免上下文过长导致进度丢失

- 你已经选择了前后台拆成两个工程
- 前台门户和后台管理的职责、风格、路由结构差异很大
- 拆开后更像长期维护的产品形态

### 2.4 建立了后端基础骨架

后端当前已具备：

- Spring Boot 启动类
- 统一返回结构 `ApiResponse`
- 全局异常处理 `GlobalExceptionHandler`
- 系统健康接口 `/api/v1/system/health`
- 项目简介接口 `/api/v1/system/profile`

为什么先做这些而不是直接写内容 CRUD：

- 因为统一返回和异常处理属于后端基础设施
- 这些结构先做好，后续写内容模块不会乱

### 2.5 建立了数据库核心表初稿

当前脚本：

- `database/schema.sql`

当前核心表：

- `ai_category`
- `ai_tag`
- `ai_source`
- `ai_content`

### 前台门户

主要作用：

- 展示首页
- 展示内容列表

### 后端

主要作用：

- 后续承接数据库、业务逻辑和数据访问

## 5. 为什么这样设计

### 为什么前端现在先做占位页

因为第一阶段重点是：

- 让工程结构成立
- 让页面入口成立
- 让目录和模块关系成立

如果一上来就塞满业务细节，很容易越写越乱。

### 为什么正文先规划为 Markdown

因为 Markdown 对这个项目很合适：

- 适合资讯、论文、项目、技术实践等内容
- 比富文本编辑器更容易解释和维护
- 后续也容易做导入、同步和渲染

## 6. 常见误区

### 误区 1：觉得第一阶段没有“业务”就没价值

错误。

第一阶段的价值在于把项目从“想法”变成“工程骨架”。
如果骨架不稳，后面的 CRUD、联调、答辩都会混乱。

### 误区 2：觉得多建几张表就是扩展性

也不对。

扩展性不等于乱加表，而是核心模型要清楚。
这也是为什么我们用 `content + category + tag + source + external_ref` 作为主干。

## 7. 老师可能会怎么问

### 问：为什么要拆前后台两个前端工程

答：

因为前台门户面向内容阅读和展示，后台面向内容运营和管理，职责差异大。
拆分后目录更清晰，也更适合长期扩展和独立演进。

### 问：为什么数据库一开始就设计来源表和外部引用表

答：

因为这个项目后续要接 GitHub、论文、公司动态等外部来源。
如果一开始不把来源和外部引用抽出来，后面接数据同步时就要重构内容表。

### 问：为什么后端先做统一返回和异常处理

答：

因为这是所有接口的公共基础。
先把返回风格和错误处理统一，后续模块开发和前后端联调会更稳定，也更容易讲清楚。
