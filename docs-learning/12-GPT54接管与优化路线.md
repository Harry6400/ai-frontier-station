# GPT-5.4 接管与优化路线

## 1. 本阶段目标

本阶段对应“阶段 10：界面设计审计与产品级视觉统一”。目标不是继续堆功能，而是让项目从“功能可用”进一步接近“真实产品”：

- 让后续模型或开发者能用更少上下文快速接管项目
- 让前台门户的视觉更克制、更高级、更适合阅读
- 让后台管理端更像内容运营控制台，而不是普通表格后台
- 保持 Vue + Spring Boot + MySQL 的主线不变
- 不破坏现有内容管理、发布、展示链路

## 3. 当前系统架构总览

当前项目是前后端分离结构：

```text
AI前沿情报站/
├─ frontend-portal/      # 前台门户：展示 AI 内容、列表、详情、项目说明
├─ frontend-admin/       # 后台管理：维护内容、分类、标签、来源
├─ backend/              # Spring Boot 接口服务
├─ database/             # MySQL 建表与初始化数据
└─ docs-learning/        # 教学、答辩、阶段进度文档
```

前台门户负责“给用户看”，重点是内容发现、内容阅读、项目定位表达。

后台管理负责“给运营者用”，重点是内容录入、编辑、发布、预览和基础数据维护。

后端负责“提供稳定接口”，通过 Controller、Service、Mapper、Entity、DTO、VO 分层把业务讲清楚。

数据库负责“内容平台的数据骨架”，核心表包括分类、标签、来源、内容、内容标签关联和外部引用预留表。

## 4. 当前已经完成的能力

当前系统已经具备：

- 前台首页真实读取精选内容、最新内容、分类和来源
- 前台内容广场支持关键词、类型、分类、来源筛选
- 前台详情页支持 Markdown 正文渲染、标签、来源链接、扩展字段
- 前台支持浅色 / 深色双主题，并会持久化主题选择
- 后台支持内容、分类、标签、来源的真实 CRUD
- 后台内容表单支持实时预览、提交前检查和三种预览模式
- 后台 Dashboard 已升级为更像内容运营控制台的结构
- 后端已经具备统一返回、分页、异常处理、内容服务和门户接口
- 数据库已经为 GitHub、论文、官方博客等外部来源预留扩展空间

## 5. 本阶段界面审计结论

### 5.1 前台门户

原有前台已经有科技感和高保真基础，但存在三个问题：

- 局部标题和卡片尺寸偏大，在较窄窗口下显得拥挤
- 装饰性图形和错位效果较多，容易压过内容本身
- 首页、列表、详情页的阅读产品感还可以进一步统一

本阶段优化方向：

- 将前台从偏“展示 Demo”收敛为偏“内容产品”
- 调整字体栈和标题尺度，减少过强的杂志感
- 统一卡片 hover、按钮、边框、阴影、浅色背景层级
- 保留浅色 / 深色双主题，同时让浅色模式更清爽
- 移动端降低标题字号和卡片圆角，避免页面显得笨重

### 5.2 后台管理端

原有后台已经收敛到 GitHub-like 风格，但还存在：

- Dashboard 信息密度偏低，更像说明页
- 筛选后缺少清晰的筛选反馈
- 移动端侧边栏占据空间较大
- 视觉上还可以更接近“内容运营控制台”

本阶段优化方向：

- 继续采用 GitHub / Primer 式中性色、细边框、低阴影
- 增加 Dashboard 的运营说明卡片和平台信号卡片
- 为内容管理页加入筛选摘要和筛选 chip
- 优化后台侧边导航、卡片、表格、弹窗和预览区层次
- 移动端让导航横向换行，减少首屏压迫感

## 6. 本阶段关键代码怎么读

### 6.1 前台设计系统

重点看：

`frontend-portal/src/styles/global.css`

这里集中定义了：

- `--font-body`：正文和普通 UI 字体
- `--font-display`：标题字体
- `--accent`：主强调色
- `--paper`、`--surface-*`：卡片和面板表面
- `--line`：边框系统
- `--shadow-soft`：统一阴影
- `[data-theme='dark']`：深色模式 token

答辩时可以这样说：

> 我没有在每个页面里随便写颜色，而是把颜色、字体、边框、阴影抽成 CSS 变量。这样前台做浅色和深色主题时，只需要切换 token，页面结构不用重写。

### 6.2 后台设计系统

重点看：

`frontend-admin/src/styles/admin.css`

这里集中定义了后台的：

- 中性色背景
- GitHub-like 边框
- 操作按钮
- 表格样式
- Dashboard 卡片
- 内容表单预览区
- 响应式布局

答辩时可以这样说：

> 前台和后台不是完全一样的视觉。前台更偏内容阅读和品牌展示，后台更偏运营效率，所以后台参考 GitHub/Primer 的克制风格，用边框、留白、信息密度来表达专业感。

### 6.3 内容管理筛选反馈

重点看：

`frontend-admin/src/views/ContentManageView.vue`

本阶段新增了：

- `activeAdminFilters`
- `filterSummaryText`
- 筛选摘要区域
- 筛选 chip 展示

这让后台不只是“能筛选”，还可以明确告诉用户当前筛选了什么、命中了多少条。

## 7. 为什么不用更复杂的方案

本阶段没有引入新的 UI 库，也没有大改后端，原因是：

- 项目当前最重要的是课程展示和可维护性
- 新增复杂依赖会增加学习成本
- 当前 Element Plus 已经满足后台表单和表格需求
- 前台用自定义 CSS 更容易形成独特视觉，而不是套模板
- 后端业务链路已经可用，现阶段不应该为了界面优化去改数据模型

## 8. 老师可能会问什么

### 问：为什么前台和后台要用不同视觉风格？

可以回答：

> 因为前台面向内容阅读者，需要强调品牌、内容层级和阅读体验；后台面向内容维护者，需要强调效率、稳定和清晰。两者风格不同，但都使用统一的边框、卡片、间距和状态反馈思想。

### 问：为什么要做设计系统？

可以回答：

> 如果每个页面单独写颜色和圆角，后续页面会越来越乱。现在把字体、颜色、阴影、边框写成 token，后续扩展 GitHub 项目、论文、公司动态栏目时，可以直接复用同一套规范。

### 问：为什么后台 Dashboard 不只是显示数字？

可以回答：

> 因为这个系统不是普通 CRUD 后台，而是 AI 内容运营平台。Dashboard 应该告诉管理员当前内容链路、来源基础和后续运营方向，所以我把它做成运营控制台雏形。

### 问：后续接 GitHub 或论文源时要怎么扩展？

可以回答：

> 数据库已经有来源表和外部引用预留表。后续可以增加采集服务，把 GitHub repo、arXiv paper 或官方博客文章同步成统一内容，再通过 content_type、source_id、extra_json 和 external_ref 保存差异信息。

## 9. 后续扩展建议

下一阶段建议不要马上做大规模采集，而是先做“小而完整”的数据源接入准备：

- 在后台来源管理中强化来源类型展示
- 在内容详情中更好展示 `extra_json`
- 增加外部引用字段的后台可视化入口
- 设计 GitHub 热门项目内容的录入模板
- 后端补充外部引用查询 VO，但暂不做自动采集

这样可以继续保持项目好讲、好跑、好维护。
