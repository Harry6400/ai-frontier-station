# 阶段9：后台同构预览增强记录

## 1. 本阶段目标

- 把后台录入台的预览从“字段摘要”升级成更像真实前台结构的同构预览。
- 让你在后台录入内容时，能够直接预演前台卡片、详情页头部和正文排版效果。
- 继续增强你对“前后台如何联动”的理解和答辩表达能力。

## 2. 为什么这一阶段很重要

上一阶段后台已经有了：

- 实时预览
- 提交前检查
- 更像 GitHub 的整体界面语言

但右侧预览还存在一个问题：

- 它更像“字段摘要面板”
- 还不够像真正前台展示结构

所以这一步的核心是：

- 不只是让后台显示数据
- 而是让后台直接预演前台展示方式

这就是“同构预览”的价值。

## 3. 这一阶段具体做了什么

### 3.1 增加三种预览模式

对应文件：

- `frontend-admin/src/views/ContentManageView.vue`

后台右侧新增了三种预览模式：

- 门户卡片
- 详情头图
- 正文排版

这样你在录入内容时，可以切换不同视角看同一条内容未来在前台的呈现方式。

### 3.2 门户卡片预览

这一步参考的是前台：

- `frontend-portal/src/components/ContentCard.vue`

现在后台右侧可以提前看到：

- 类型标签
- 日期
- 英文副标题
- 标题
- 摘要
- 分类 / 来源
- 标签
- “阅读全文”按钮区域

这能帮助你提前判断：

- 标题是否过长
- 摘要是否适合列表卡片
- 标签数量是否合适

### 3.3 详情头图预览

这一步参考的是前台：

- `frontend-portal/src/views/ContentDetailView.vue`

后台现在可以预演详情页头部的核心结构，包括：

- 类型 + 分类
- 标题
- 摘要
- 发布时间
- 来源
- 作者
- 阅读时长
- 标签

这会让“后台录入”和“前台详情展示”之间的关系更清楚。

### 3.4 正文排版预览

正文预览不再只是原始文本。

我补了一个轻量 Markdown 预览层，可以显示：

- 一级/二级/三级标题
- 段落
- 列表
- 行内代码
- 代码块

这让后台可以更接近前台正文区的阅读结构。

## 4. 这一阶段为什么不直接复用前台 `marked`

这是一个很值得讲的点。

当前后台没有直接引入前台那套 `marked` 依赖，而是先做了一个轻量 Markdown 预览器，原因是：

- 当前阶段优先保证录入体验和展示结构跑通
- 避免为了一个预览功能立刻增加新的依赖复杂度
- 先做结构一致，再考虑把渲染实现进一步统一

也就是说，这一步体现的是：

- 先把系统关系讲清楚
- 再决定要不要做完全共用实现

## 5. 这一阶段关键代码怎么理解

### 5.1 `previewMode`

这是右侧预览切换的核心状态。

它的作用是：

- 决定当前展示卡片预览、详情预览还是正文预览

这说明后台预览不是固定死的一张面板，而是一个可切换的预览系统。

### 5.2 `renderMarkdownPreview()`

这是本阶段最值得读的函数之一。

它做的事情是：

- 先转义 HTML，避免直接把原始输入当成 HTML 执行
- 再识别 Markdown 常见结构
- 最后拼出适合预览的 HTML

你不需要把它理解成完整 Markdown 解析器。

你只需要记住：

- 当前版本是为了后台预览做的“轻量结构化渲染”
- 目标是帮助录入时确认内容层次，不是完全替代前台渲染器

### 5.3 模板中的三段预览结构

你可以重点看模板里这三块：

- `previewMode === 'card'`
- `previewMode === 'detail'`
- `previewMode === 'article'`

它们分别对应三种前台展示场景。

这是一种很典型的前后端思路：

- 一份内容数据
- 多种前台展示视图
- 后台录入时提前模拟这些视图

## 6. 这一步适合怎么答辩

### 6.1 如果老师问：为什么后台还要做这么复杂的预览？

你可以答：

因为这是内容平台，不是普通表单系统。内容录入完成后最终要进入前台展示，所以后台应该帮助操作者提前判断卡片效果、详情头图效果和正文阅读效果。

### 6.2 如果老师问：同构预览是什么意思？

你可以答：

这里的意思不是后台和前台完全共用同一个组件，而是后台预览尽量贴近前台真实展示结构，让录入阶段就能预演展示效果。

### 6.3 如果老师问：为什么正文预览没完全复用前台渲染？

你可以答：

因为当前阶段优先保证系统结构清晰和功能完整，先用轻量预览方案把录入体验做起来。后续如果继续完善，可以再把后台正文预览和前台渲染进一步统一。

## 7. 常见误区

- 误区 1：后台预览只要能看到文字就够了
  实际上内容平台更需要看到“展示结构”，而不只是字段值。

- 误区 2：同构预览一定要完全复制前台代码
  当前更重要的是先保持展示逻辑一致，再逐步考虑代码实现是否合并。

- 误区 3：正文预览必须一次做到完整 Markdown 编辑器
  对课程项目来说，先做轻量结构化预览是更合理的节奏。

## 8. 本阶段之后，系统到了什么程度

到这一步为止，这个项目的后台已经不再只是录入系统，而更像一个真正的内容运营台：

- 可以录入
- 可以校验
- 可以预览
- 可以模拟前台展示结构

这意味着你在课堂上可以直接演示完整链路：

1. 后台创建内容
2. 切换不同预览模式查看效果
3. 保存
4. 到前台查看实际展示

这个演示会非常有说服力。

## 9. 下一阶段建议

下一步最适合继续做两类事情之一：

- 方案 A：补更强的表单联动
  比如根据内容类型自动提示字段重点、自动建议阅读时长、自动建议来源类型

- 方案 B：开始为未来聚合平台扩展做准备
  比如增加外部引用字段展示、同步状态占位、采集模块骨架

如果优先考虑“答辩可讲性”，我建议先做方案 A。
