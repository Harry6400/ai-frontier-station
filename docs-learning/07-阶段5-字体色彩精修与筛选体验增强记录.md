# 阶段5：字体色彩精修与筛选体验增强记录

## 1. 本阶段目标

- 让前台门户的字体和颜色不只是“好看一点”，而是更接近真正 AI 内容产品的高级感。
- 把内容广场从“能筛选”升级成“筛选体验完整”的页面。
- 在不推倒现有结构的前提下，继续强化第一版可展示、可答辩、可扩展的特性。

## 2. 这一阶段具体做了什么

### 2.1 门户设计系统再精修

本阶段继续以 `huashu-design` 的思路为主：

- 不走通用蓝紫色 SaaS 套路
- 不把页面做成普通后台系统的附属首页
- 强调“编辑型内容平台 + 科技感”的组合气质

实际落地在：

- `frontend-portal/src/styles/global.css`

核心调整包括：

- 引入 `--font-body`、`--font-display`、`--font-mono` 三层字体变量
- 把大标题、内容标题、正文代码块分别放到更合适的字体语境里
- 重做浅色 / 深色两套主色 token，让背景、卡片、强调色、描边统一起来
- 把按钮、标签、卡片、输入框、状态面板都改成同一套 token 驱动

## 3. 为什么这样改

### 3.1 为什么要拆成正文体、展示体、等宽体

如果全站都用一套无衬线字体，页面会显得比较“平”，像工程页面多于内容平台。

所以这里采用：

- 正文字体：保证阅读效率和界面一致性
- 展示字体：给首页标题、详情标题、内容卡片标题更强的产品气质
- 等宽字体：用于代码块和技术内容细节展示

这样答辩时你可以说：

- 我不是单纯换了一个字体，而是做了字体层级设计。
- 这是内容平台常见的做法，因为“信息阅读”和“品牌气质表达”不完全是同一件事。

### 3.2 为什么颜色不是越亮越科技

很多初学者会把“科技感”理解成高饱和蓝色、紫色、大面积发光。
这样很容易廉价。

这一版改法是：

- 背景保持偏暖的纸感和雾面感
- 强调色收敛成更稳的冷蓝
- 深色模式不做纯黑，而是做深海军蓝
- 用层次和透明度做高级感，不靠刺眼颜色堆出来

## 4. 内容广场这一阶段为什么值得讲

### 4.1 增加了真正的来源筛选

前台现在不仅能按：

- 关键词
- 内容类型
- 分类

还可以按：

- 来源

来源数据来自后端真实接口，不是前端写死。

对应代码：

- 后端返回结构：`backend/src/main/java/com/harry/aifrontier/vo/HomeOverviewVO.java`
- 后端首页聚合：`backend/src/main/java/com/harry/aifrontier/service/impl/ContentServiceImpl.java`
- 前台列表页：`frontend-portal/src/views/ContentListView.vue`

### 4.2 筛选体验不只是多一个下拉框

这一步还补了：

- 当前筛选条件展示
- 点击条件即可移除
- 结果摘要
- 无结果空状态
- 清空筛选按钮

这样页面更像真实产品，而不是课程作业里常见的“几个输入框 + 一个查询按钮”。

## 5. 这一阶段涉及的关键文件怎么读

### 5.1 `frontend-portal/src/styles/global.css`

重点看三部分：

- 顶部 `:root` 和 `:root[data-theme='dark']`
  这里是整套设计 token 的核心。
- 标题字体相关选择器
  这里能看出哪些内容用展示字体，哪些保持正文风格。
- `filter-*`、`active-filter-*`、`empty-state-panel`
  这里是内容广场筛选体验增强的主要样式。

### 5.2 `frontend-portal/src/views/ContentListView.vue`

重点看：

- `filters`：前台筛选状态
- `activeFilters`：把当前筛选条件转成可展示的标签
- `resultSummary`：把分页和筛选状态转成用户能理解的话
- `loadFilterOptions()`：从后端取分类和来源

答辩时你可以说：

- 页面不是直接把接口结果堆上去，而是做了一层“前端展示状态整理”。
- 这样页面可读性更好，后续扩展筛选条件也更方便。

### 5.3 `backend/src/main/java/com/harry/aifrontier/service/impl/ContentServiceImpl.java`

这一阶段重点不是重写查询逻辑，而是：

- 复用已有门户首页接口
- 在首页聚合结果里继续补充 `sources`

这样做的好处是：

- 改动小
- 不破坏现有架构
- 前台需要的筛选数据有了统一来源

## 6. 为什么不另外新建一个“来源选项接口”

这是一道老师很可能会追问的问题。

当前我的选择是：

- 第一版优先复用 `portal/home`
- 因为首页本来就是门户的聚合入口
- 现在把分类、精选内容、最新内容、来源一起放进去，复杂度可控

为什么暂时不单独拆接口：

- 第一版数据量小，没必要过早拆太细
- 对新手来说，一个聚合入口更容易理解
- 以后如果首页数据越来越重，再拆成独立的 `portal/options` 接口也不晚

也就是说，这里体现的是：

- 当前阶段先保持简单
- 但结构上没有封死扩展路线

## 7. 常见误区

- 误区 1：觉得前端优化就是换个配色
  实际上真正重要的是字体层级、间距、表面层次、状态反馈是否统一。

- 误区 2：觉得筛选功能就是把字段传给接口
  实际上用户还需要知道自己筛了什么、结果为什么变少、没有结果时怎么办。

- 误区 3：为了“未来扩展”一开始就拆很多接口和模块
  这会让第一版太难做，也不利于你答辩时讲清楚。

## 8. 老师可能怎么问

### 8.1 为什么前台列表页要做这么多筛选状态处理？

你可以答：

因为内容平台不是普通 CRUD 页面。用户在找内容时，需要清楚知道当前筛选条件、结果数量和页面状态，所以我在前端增加了筛选反馈层，而不是只保留原始接口字段。

### 8.2 为什么首页接口里要返回来源列表？

你可以答：

第一版优先保证结构清晰和接口数量可控。门户首页本身就是聚合接口，所以我把来源列表也放进去，前台筛选可以直接复用。后续如果来源选项越来越复杂，再单独拆接口。

### 8.3 为什么标题不用和正文同一种字体？

你可以答：

因为内容平台既要承担阅读功能，也要承担品牌展示功能。正文优先清晰易读，标题优先建立视觉气质，所以我做了字体层级拆分。

## 9. 后续怎么扩展

这一阶段的改动为后续留了几个很自然的扩展点：

- 列表页可以继续加标签筛选、排序方式、热门度筛选
- 首页聚合接口后续可以拆分成首页内容接口和筛选项接口
- 设计 token 后续可以继续扩到后台管理端
- 如果以后接 GitHub、论文、公司博客采集，来源筛选可以直接复用当前模型

## 10. 本阶段结论

这一阶段不是“又加了几个样式细节”，而是做了两件对项目气质很关键的事：

- 把前台门户从普通课程页面进一步拉向真正的内容产品
- 把内容广场从基础列表页拉向可讲、可用、可扩展的检索入口

这正好符合项目的双目标：

- 现在更适合展示和答辩
- 以后更适合继续扩展成 AI 信息聚合平台
