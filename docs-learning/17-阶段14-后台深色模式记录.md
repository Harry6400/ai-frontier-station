# 阶段 14：后台深色模式记录

## 1. 本阶段目标

本阶段给后台管理端补充浅色 / 深色双主题能力，让后台在视觉上也形成完整的产品体验。

本阶段只做后台深色模式，不改业务逻辑，不做登录权限，不处理自动采集，也不顺手做包体优化。

## 2. 为什么现在做后台深色模式

前台门户已经支持浅色 / 深色模式，但后台管理端此前只有浅色模式。这样在答辩演示时，前台和后台的产品完整度不一致。

后台补齐深色模式后，可以讲清楚三个点：

- 前台和后台虽然面向不同用户，但都属于同一个产品体系。
- 主题切换不是复制两套页面，而是通过设计 token 统一控制颜色。
- 后台保持 GitHub / Primer 式的克制工程感，不做廉价的高对比霓虹风。

## 3. 本阶段涉及的技术点

- Vue 3 组件状态初始化
- Pinia 全局状态管理
- `localStorage` 本地持久化
- `prefers-color-scheme` 系统主题偏好读取
- `document.documentElement.dataset` 写入主题标记
- CSS 变量，也就是设计 token
- Element Plus 常用组件的暗色适配

## 4. 当前代码结构怎么理解

本阶段主要改动三个后台文件：

```text
frontend-admin/
├─ src/
│  ├─ App.vue
│  ├─ stores/
│  │  └─ useAdminThemeStore.js
│  └─ styles/
│     └─ admin.css
```

`useAdminThemeStore.js` 负责保存后台主题状态。

`App.vue` 负责在后台壳层初始化主题，并在顶部 header 提供切换按钮。

`admin.css` 负责定义浅色和深色两套颜色 token，并覆盖后台页面、表格、表单、弹窗、分页、标签等样式。

## 5. 为什么用 CSS token，而不是复制两套页面

如果复制两套后台页面，短期看似简单，但后续维护会非常麻烦。例如内容管理页有一个按钮要改，就必须同时改浅色版和深色版，容易漏改。

本项目采用的方式是：

```css
:root {
  --admin-bg: #f6f8fa;
  --admin-card: #ffffff;
}

:root[data-admin-theme='dark'] {
  --admin-bg: #0d1117;
  --admin-card: #161b22;
}
```

页面结构只写一套，颜色通过变量切换。这更符合真实项目里的设计系统思路，也更适合答辩时解释“可维护性”。

## 6. 关键代码应该怎么读

### 6.1 主题 Store

`frontend-admin/src/stores/useAdminThemeStore.js` 中的核心职责是：

- 首次进入时读取浏览器本地存储
- 如果本地没有保存过主题，就跟随系统主题
- 切换主题时写入 `localStorage`
- 把主题写到 `document.documentElement.dataset.adminTheme`

这样 CSS 就能通过 `[data-admin-theme='dark']` 判断当前主题。

### 6.2 后台壳层

`frontend-admin/src/App.vue` 中新增主题初始化和切换按钮。

后台 header 里保留了“查看前台门户”按钮，同时增加“浅色 / 深色模式”切换按钮。这样老师演示时可以直接在后台切换主题，不需要改代码。

### 6.3 后台样式

`frontend-admin/src/styles/admin.css` 中新增了深色模式 token：

- 页面背景
- 侧边栏背景
- 卡片背景
- 边框颜色
- 主文本和次级文本
- 品牌色
- 表格头部
- hover 状态
- 输入框、弹窗、分页、标签等 Element Plus 组件

这说明本项目不是简单把背景改黑，而是按设计系统方式处理不同 UI 层级。

## 7. 为什么后台风格不做成强烈科技霓虹风

后台管理端的主要任务是录入、编辑、筛选、维护数据。它需要稳定、可信、可长时间使用。

所以后台参考 GitHub / Primer 的设计方向：

- 中性色为主
- 细边框而不是厚重阴影
- 清晰层级而不是强烈装饰
- 按钮和状态色保持克制

这和前台门户的内容展示气质不同，但二者共享“高级、清晰、可扩展”的产品目标。

## 8. 后台包体偏大是什么意思

构建后台时，Vite 会提示某个 JavaScript chunk 超过 500KB。

这不是运行错误，也不是页面坏了。它的含义是：后台当前一次性打包了较多代码，首次加载时浏览器需要下载更大的 JS 文件。

当前主要原因是后台使用了 Element Plus，并且采用了整体引入方式：

```js
import ElementPlus from 'element-plus'
import 'element-plus/dist/index.css'
```

这对新手阶段更容易理解，也方便快速完成课程项目。后续如果要优化，可以再做：

- Element Plus 按需引入
- 路由懒加载
- Vite 手动分包
- 图标按需加载

本阶段按计划不处理包体优化，所以构建警告仍然保留。

## 9. 常见误区

- 误区一：深色模式就是把背景改成黑色。
- 真实做法：要同时处理文本、边框、卡片、表单、弹窗、表格和状态色。

- 误区二：深色模式应该复制一套页面。
- 真实做法：页面结构复用，颜色用 token 切换。

- 误区三：后台也要和前台一样炫。
- 真实做法：后台应该更克制，因为它强调效率、稳定和可维护。

## 10. 老师可能会怎么问

### 问：为什么后台也要做深色模式？

可以回答：前台已经支持双主题，如果后台没有主题系统，产品体验会不完整。后台深色模式能展示我们有统一设计系统意识，并且可以说明前后台虽然风格不同，但仍然使用同一类 token 化设计方法。

### 问：为什么用 Pinia 管理主题？

可以回答：主题属于跨组件共享状态。虽然现在只在 App 顶栏切换，但后续其他后台页面或组件也可能需要读取当前主题，所以放到 Pinia 更清晰。

### 问：为什么用 CSS 变量？

可以回答：CSS 变量可以让同一套页面结构在不同主题下复用，只切换颜色 token，不需要复制组件。这提高了可维护性，也方便后续继续增加主题或统一设计规范。

### 问：为什么不现在优化包体？

可以回答：当前阶段目标是补齐后台主题体验，不扩大范围。包体优化属于性能优化阶段，会涉及 Element Plus 按需引入、路由懒加载和分包策略，后续可以单独作为一阶段做，避免本阶段同时改太多东西影响稳定性。
