# 阶段 11：外部数据源接入准备记录

## 1. 本阶段目标

本阶段不是做真正的自动采集，而是先把“未来要接 GitHub、论文、公司动态、技术社区”这件事，在当前系统里做成可展示、可解释、可继续扩展的结构。

换句话说：

- 现在仍然保持人工录入和后台管理主流程
- 但后台开始知道不同来源类型意味着什么
- 内容录入时可以按内容类型生成扩展字段模板
- 前台详情页可以解释这些扩展字段未来会如何用于采集、同步和推荐

这样做的好处是：第一版不会过度复杂，但后续接采集模块时不用推倒重来。

## 2. 本阶段涉及的技术点

- Vue 3 的 `computed` 用于根据来源类型和内容类型动态生成展示信息
- Element Plus 的表格、标签、选择器和表单继续用于后台管理
- 后端 VO 增加字段，让前台和后台能拿到 `sourceType`
- JSON 扩展字段继续保存在 `ai_content.extra_json`
- 前台详情页对扩展字段做可读化展示，而不是只显示原始 JSON

后台内容管理页负责提供不同内容类型的扩展字段模板。

前台详情页负责把扩展字段转换成用户和老师能看懂的信息。

## 4. 为什么这样设计

项目长期目标是 AI 信息聚合平台，所以未来一定会遇到多种外部来源：

- GitHub 仓库
- arXiv 论文
- 官方博客
- 技术社区文章
- 人工精选内容

这些来源的数据字段不完全一样。如果一开始就为每种来源单独建一套大表，项目会很复杂，不适合课程阶段。

所以当前采用折中方案：

- 通用内容字段放在 `ai_content`
- 来源统一放在 `ai_source`
- 标签统一放在 `ai_tag`
- 差异化外部字段先放在 `extra_json`
- 后续更复杂的同步关系放到 `ai_content_external_ref`

这样第一版能讲清楚，未来也有扩展余地。

## 5. 为什么不用真正自动采集

自动采集涉及：

- 第三方 API
- 定时任务
- 数据清洗
- 去重
- 失败重试
- 审核流程
- 内容安全

这些都很有价值，但对于当前课程项目来说会显著增加复杂度。

所以当前先做“采集前置结构”：

- 来源类型可维护
- 内容类型可区分
- 外部字段可录入
- 前台可以展示映射关系

等这个基础稳定后，再接 GitHub API 或论文 API 才更稳。

## 6. 关键代码应该怎么读

### 6.1 后端如何返回来源类型

看 `ContentServiceImpl.java` 的 `toListItemVO` 和 `toDetailVO`。

当内容绑定了来源时，后端不只返回来源名称，也返回来源类型：

```java
vo.setSourceName(source.getName());
vo.setSourceType(source.getSourceType());
```

答辩时可以说：

> 来源名称用于展示，来源类型用于判断它属于 GitHub、论文站点、官方博客还是人工录入。这样前端和后续采集模块都可以根据 sourceType 做不同处理。

### 6.2 后台来源类型卡片

看 `SourceManageView.vue`。

页面中定义了 `sourceTypeOptions`，包含：

- `github`
- `paper`
- `official_blog`
- `community`
- `manual`

每种来源都有说明和数量统计。

答辩时可以说：

> 来源管理不是简单存一个名字，而是为未来多数据源接入做注册表。以后接 GitHub 或论文源时，可以先在来源表注册，再由采集模块同步内容。

### 6.3 内容扩展字段模板

看 `ContentManageView.vue` 的 `EXTENSION_TEMPLATES`。

不同内容类型有不同模板：

- 热门项目：`github_repo`、`githubRepo`、`starTrend`、`language`
- 论文：`arxiv_paper`、`arxivId`、`paperVenue`、`projectPage`
- 公司动态：`official_post`、`companyFocus`、`productLine`
- 技术实践：`community_practice`、`practiceScope`、`difficulty`

答辩时可以说：

> 当前不是自动采集，但我先把未来采集后的字段结构预留出来。比如 GitHub 项目后续可以同步仓库名、语言、Star 趋势；论文可以同步 arXiv ID 和会议来源。

### 6.4 前台详情页如何解释扩展字段

看 `frontend-portal/src/utils/content.js` 和 `ContentDetailView.vue`。

`parseExtraJson` 不再只返回原始键值，而是返回：

- 字段 key
- 字段 value
- 中文 label
- 字段说明 note

这样前台详情页可以把扩展字段展示为“后续采集字段”，而不是一串难懂 JSON。

## 7. 常见误区

### 误区 1：现在没做自动采集，就不算可扩展

不是。可扩展首先体现在数据模型和模块边界上。当前已经有来源表、内容类型、扩展 JSON 和外部引用表，这些就是后续采集的基础。

### 误区 2：所有字段都应该建成数据库列

不一定。对于课程第一版，通用字段建列，差异字段放 JSON 更灵活。等某些字段稳定后，再单独建列也可以。

### 误区 3：前台不应该展示扩展字段

前台展示扩展字段不是为了普通用户看数据库，而是为了课程答辩和产品解释：让老师看到这个系统确实为未来 AI 信息聚合做了准备。

## 8. 老师可能会怎么问

### 问：为什么要有 sourceType？

可以回答：

> 因为来源不仅有名称，还有类型。GitHub、论文站点、官方博客的数据结构不同，sourceType 可以帮助系统判断未来该用什么采集逻辑和展示逻辑。

### 问：extra_json 是不是偷懒？

可以回答：

> 不是。通用字段已经建成数据库列，例如标题、摘要、来源、分类、标签、正文。extra_json 用来保存不同外部来源的差异化字段，避免第一版为了每种来源建很多复杂表。

### 问：以后真的接 GitHub 怎么做？

可以回答：

> 可以新增一个 GitHub 同步服务，定时读取 GitHub Trending 或指定仓库信息，然后把仓库名、语言、Star 趋势写入 extra_json，把原始链接写入 source_url，把外部 ID 写入 ai_content_external_ref。

### 问：为什么现在先做模板？

可以回答：

> 模板可以帮助后台录入人员按照未来采集字段的结构录入内容。这样现在是人工维护，未来换成自动采集时字段结构已经比较稳定。
