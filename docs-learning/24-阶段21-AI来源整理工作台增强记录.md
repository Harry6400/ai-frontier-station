# 阶段21：AI 来源整理工作台增强记录

## 1. 本阶段目标

阶段20已经完成“人工来源输入 + 百炼总结 + 创建草稿”的基本闭环。本阶段继续增强后台体验，让它更像一个真正的 AI 内容运营工作台，而不是一个简单接口测试弹窗。

本阶段重点：

1. 给不同来源类型提供模板。
2. 把 AI 标签建议匹配到已有标签。
3. 保持未匹配标签为建议，不自动写入标签库。
4. 支持 AI 结果直接创建草稿，也支持回填到普通内容编辑表单继续人工修改。
5. 优化百炼调用失败时的提示。

## 2. 本阶段涉及的技术点

- Vue 组合式 API 的状态组织
- 后台表单模板与默认值填充
- 前端标签匹配与去重
- AI 结果人工确认流程
- 后端外部 API 错误兜底
- 不新增数据库表的渐进式增强

## 3. 当前代码结构怎么理解

后台主要改动集中在 `frontend-admin/src/views/ContentManageView.vue`：

- `AI_SOURCE_TEMPLATES`：定义结构化榜单源、官方动态源、GitHub 项目源、论文研究源、社区实践源、人工录入源。
- `applyAiSourceTemplate`：点击模板后填充来源类型、内容类型、来源名称、摘要提示和整理要求。
- `matchSuggestedTags`：把百炼返回的 `tagSuggestions` 和现有标签按名称匹配。
- `fillContentFormFromAiSource`：把 AI 结果回填到普通内容编辑表单，方便继续人工修改。
- `submitAiSourceImport`：直接创建草稿，仍然不自动发布。

后端主要改动在 `AiSummaryServiceImpl`：

- 未配置 `DASHSCOPE_API_KEY` 时直接返回明确提示。
- 原文摘要过短时拒绝生成，避免 AI 胡编。
- 百炼访问失败时返回统一提示，不暴露底层异常和密钥信息。

## 4. 为什么这样设计

AI 生成内容不能完全自动进入发布状态，因为外部信息可能不完整，AI 也可能总结偏差。因此本阶段保留两条人工确认路径：

1. **创建草稿**：适合快速演示“AI 总结层生成内容”。
2. **回填编辑表单**：适合继续人工修改标题、摘要、正文、标签和扩展 JSON。

标签建议不自动创建数据库标签，是为了避免标签库被 AI 生成的近义词污染。例如“LLM”“大模型”“大语言模型”可能都指向同一类内容，如果自动创建，会让后续筛选和答辩解释变乱。第一版更适合让管理员确认。

## 5. 为什么不用别的方案

没有直接做真实网页抓取，因为那会引入反爬、版权、网页解析、失败重试等复杂问题。

没有直接做百炼联网搜索，因为当前项目更需要先把“来源管理 + 人工确认 + AI 总结层”的平台结构讲清楚。

没有新增标签自动创建接口，因为标签是内容平台的重要分类资产，第一版应该保持可控。

## 6. 关键代码应该怎么读

先读后台模板：

```js
const AI_SOURCE_TEMPLATES = { ... }
```

它说明不同数据源不是混在一起的，而是通过模板把输入结构先规范起来。

再读标签匹配：

```js
function matchSuggestedTags(suggestions) { ... }
```

它只匹配已有标签，不创建新标签，体现“AI 给建议，人来确认”的设计边界。

最后读回填逻辑：

```js
function fillContentFormFromAiSource() { ... }
```

它把 AI 结果转成普通内容编辑表单里的字段，说明这个工作台不是孤立功能，而是复用原有内容管理能力。

## 7. 常见误区

- 不要让 AI 结果自动发布。
- 不要把未匹配标签直接写进数据库。
- 不要把 API Key 放到前端。
- 不要把来源模板理解为采集器，模板只是规范人工输入。
- 不要全文复制原文，仍然只保存摘要、摘录和原始链接。

## 8. 老师可能会怎么问

**问：为什么要有来源模板？**

答：不同来源的信息结构不同。榜单源关注指标和排名，GitHub 源关注仓库、语言和 Star，论文源关注方法和实验结果。模板可以让后台输入更规范，也方便后续接自动采集。

**问：为什么不自动创建标签？**

答：标签会影响筛选和内容组织，如果完全由 AI 自动创建，容易出现重复标签和同义标签。第一版只做匹配和建议，由管理员确认更安全。

**问：为什么要有“回填到编辑表单”？**

答：它体现了人工审核流程。AI 先生成草稿信息，管理员可以继续修改标题、摘要、正文和标签，避免 AI 内容未经审核直接进入系统。

**问：这个阶段和未来采集模块有什么关系？**

答：未来自动采集模块可以替代人工输入来源信息，但后面的 AI 总结、标签建议、草稿确认、外部引用记录这些流程都可以复用。

## 9. 后续如何扩展

下一阶段可以开始做“真实数据源接入前置设计”，建议先在两个方向中选一个：

1. GitHub API：结构化、稳定、容易演示，适合把手动 GitHub 导入升级为真实仓库查询。
2. 百炼联网搜索：更接近“按指定方法搜索网上一手信息”，但来源稳定性、可解释性和费用控制要先设计好。

如果优先保证课堂可展示，建议先做 GitHub API；如果优先贴近长期愿景，建议先规划百炼联网搜索。
