# 阶段20：数据源分层与百炼总结接入记录

## 1. 本阶段目标

本阶段不是直接做“全网自动搜索”，而是先把未来 AI 信息聚合平台的数据源体系拆清楚，并完成一个可演示的最小闭环：

1. 后台人工录入来源标题、链接、来源类型、原文摘要和关键摘录。
2. 后端读取环境变量 `DASHSCOPE_API_KEY` 调用阿里百炼 OpenAI 兼容接口。
3. 百炼生成 AI 导读、推荐理由、重要性评分、标签建议和正文 Markdown。
4. 后台展示生成结果，管理员确认后创建草稿内容。
5. 前台详情页展示“AI 导读在上，来源/原文摘录在下”的结构。

这样做的重点是：百炼是“AI 总结层”，不是唯一数据源；真正的数据源仍然来自官方博客、榜单、GitHub、论文站点和社区。

## 2. 本阶段涉及的技术点

- Spring Boot 后端调用外部 HTTP API
- 后端环境变量读取：`DASHSCOPE_API_KEY`
- OpenAI 兼容 Chat Completions 接口
- DTO / VO 分层
- `extra_json` 作为过渡扩展字段
- 前端后台弹窗表单与确认流程
- 前台详情页根据扩展 JSON 做差异化展示
- 不全文转载原文，只保存摘要、摘录、链接和图片地址

## 3. 当前代码结构怎么理解

## 4. 为什么这样设计

第一版不直接做爬虫，因为爬虫涉及反爬、版权、网页结构解析、失败重试、定时任务和数据去重，范围会迅速变大。现在先做“人工提供来源 + 百炼总结”，有三个好处：

1. 课堂演示能讲清楚完整链路。
2. API Key 只在后端使用，安全边界清楚。
3. 未来接入 GitHub API、论文 API、榜单数据时，可以复用同一套 AI 总结层。

数据库暂时不新增表，而是把 AI 导读、推荐理由、来源说明、原文摘录等信息放进 `ai_content.extra_json`。这是一个过渡设计：第一版简单，后续如果数据量变大，可以再拆成 `source_snapshot`、`ai_summary` 等独立表。

## 5. 为什么不用“全部用百炼搜”

全部依赖百炼搜索会导致来源结构不稳定，也不利于答辩解释。更合理的架构是分层：

- 结构化榜单源：Arena、Artificial Analysis、HELM、Hugging Face Leaderboard、MTEB、SWE-bench
- 官方动态源：OpenAI、Anthropic、Google DeepMind、Meta AI、Mistral、xAI、Hugging Face 官方博客和 X
- 社区热度源：GitHub Trending、Hugging Face Trending、Papers with Code、X 关键账号
- AI 总结层：百炼负责总结、分类、打标签、生成推荐理由
- 数据库存储层：保存原始链接、来源、标题、时间、摘要、AI 总结、标签、重要性评分

## 6. 关键代码应该怎么读

先读 `AiSummaryServiceImpl#summarizeSource`：

1. 判断是否配置 `DASHSCOPE_API_KEY`。
2. 调用 `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`。
3. 要求模型返回严格 JSON。
4. 解析为 `AiSourceSummaryVO`。

再读 `ContentServiceImpl#importAiSource`：

1. 把 AI 结果转成 `ContentSaveRequest`。
2. 创建一条 `DRAFT` 内容，避免 AI 结果自动发布。
3. 创建一条 `ai_content_external_ref`，记录原始来源链接和快照。

最后读 `ContentDetailView.vue`：

1. 解析 `detail.extraJson`。
2. 如果存在 `aiSummary`、`recommendationReason`、`originalExcerpt` 等字段，就展示 AI 导读面板。
3. 正文仍然走 Markdown 安全清洗，不直接信任外部内容。

## 7. 常见误区

- 不要把百炼 API Key 写到前端代码。
- 不要把 AI 生成内容直接发布，应该先创建草稿。
- 不要全文复制原文，优先保存摘要、摘录和原始链接。
- 不要把百炼当成唯一数据源，它更适合作为总结和结构化层。
- 不要一开始就做复杂爬虫，课程项目要优先保证可运行、可解释、可扩展。

## 8. 老师可能会怎么问

**问：为什么不直接做自动采集？**

答：自动采集涉及网页解析、反爬、版权、定时任务和失败重试，第一版先做人工来源输入和 AI 总结，可以完整展示内容平台的核心闭环，后续再逐步替换数据输入层。

**问：百炼 API Key 为什么放后端？**

答：API Key 属于服务端密钥，放前端会暴露给浏览器用户。后端统一调用也方便做限流、日志、异常处理和权限控制。

**问：为什么 AI 字段先放 `extra_json`？**

答：因为第一版字段仍在探索期，用 JSON 可以避免频繁改表；当字段稳定后，再拆成独立表，这是从演示项目平滑演进到长期项目的折中设计。

**问：为什么 AI 生成后只创建草稿？**

答：AI 结果需要人工确认，尤其是外部信息来源和摘要。创建草稿可以避免未审核内容直接出现在前台。

## 9. 后续如何扩展

下一步可以做“AI 来源整理工作台增强”：

- 保存 AI 生成失败原因和重试提示
- 支持从 URL 自动提取标题和站点名
- 支持把标签建议匹配到已有标签
- 支持选择结构化来源模板，如榜单源、官方博客、GitHub 项目、论文源
- 后续再接真实 GitHub API、论文数据源或百炼联网搜索能力

更远期可以拆表：

- `ai_source_snapshot`：保存原始来源快照
- `ai_summary_result`：保存不同模型生成的摘要结果
- `ai_ingestion_job`：保存采集任务和同步状态
