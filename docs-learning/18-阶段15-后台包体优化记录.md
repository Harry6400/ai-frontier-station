# 阶段 15：后台包体优化记录

## 1. 本阶段目标

本阶段优化后台管理端的前端构建包体，解决 Vite 构建时提示 `Some chunks are larger than 500 kB` 的问题。

本阶段只处理后台构建体积，不改业务接口、不改页面功能、不改深色模式逻辑。

## 2. 为什么要做包体优化

后台之前使用了 Element Plus 整包引入：

```js
import ElementPlus from 'element-plus'
import 'element-plus/dist/index.css'
```

这种写法对新手友好，但会把大量当前没用到的组件和样式一起打进构建结果。后台页面不多时，这会让首屏 JS 文件偏大。

优化后，项目可以在答辩中说明：第一版不是只追求“能跑”，也开始考虑真实项目里的加载性能和工程结构。

## 3. 本阶段涉及的技术点

- Element Plus 手动按需注册
- Element Plus 子路径导入
- Element Plus 组件样式按需引入
- Vue Router 路由懒加载
- Vite / Rollup `manualChunks` 分包
- 构建产物体积分析

## 4. 当前代码结构怎么理解

本阶段主要改动三个后台工程文件：

```text
frontend-admin/
├─ src/
│  ├─ main.js
│  └─ router/
│     └─ index.js
└─ vite.config.js
```

`main.js` 负责后台应用入口和 Element Plus 组件注册。

`router/index.js` 负责后台页面路由，本阶段改为按页面懒加载。

`vite.config.js` 负责构建配置，本阶段新增依赖分包策略。

## 5. 为什么不用自动按需导入插件

自动按需导入插件例如 `unplugin-vue-components` 可以减少手写注册代码，但会新增构建插件和额外配置。

当前项目是课程展示 + 新手可答辩项目，所以本阶段选择手动按需注册：

- 代码更直观
- 不新增依赖
- 更容易向老师解释
- 当前后台使用的 Element Plus 组件数量不多

代价是后续如果新增新的 `el-*` 组件，需要手动在 `main.js` 中注册并导入对应样式。

## 6. 关键代码应该怎么读

### 6.1 Element Plus 按需注册

`frontend-admin/src/main.js` 不再使用 `.use(ElementPlus)` 整包注册，而是只注册实际用到的组件。

例如：

```js
import { ElButton } from 'element-plus/es/components/button/index'
import 'element-plus/theme-chalk/el-button.css'
```

这表示：只引入按钮组件和按钮样式，不把整个 Element Plus 都打进入口。

### 6.2 路由懒加载

`frontend-admin/src/router/index.js` 中页面组件改为：

```js
component: () => import('../views/ContentManageView.vue')
```

这样用户首次进入后台时，不需要一次性加载所有管理页面。访问某个页面时，浏览器再加载对应页面 chunk。

### 6.3 Vite 分包

`frontend-admin/vite.config.js` 中通过 `manualChunks` 把依赖拆成：

- `vue-vendor`：Vue、Vue Router、Pinia
- `element-plus`：Element Plus 组件相关代码
- `admin-vendor`：axios 等后台通用依赖
- `vendor`：其他第三方依赖

这样构建结果更清晰，也方便观察包体来源。

## 7. 优化前后对比

优化前后台构建结果中，主要 JS 文件约为：

```text

```

优化后后台构建结果中，主要 JS 文件被拆分为：

```text
dist/assets/element-plus-fVy3pnha.js        340.77 kB
dist/assets/vendor-DiQ2czE1.js              180.54 kB
dist/assets/ContentManageView-3ivUij5o.js    30.07 kB
dist/assets/vue-vendor-DkJmWqNY.js           29.43 kB
dist/assets/index-C08cORb8.js                 6.91 kB
```

本阶段优化后，后台构建不再出现 500KB chunk 警告。

## 8. 常见误区

- 误区一：包体优化就是删功能。
- 真实做法：本项目没有删除功能，只改变代码加载方式。

- 误区二：按需注册一定要用插件。
- 真实做法：组件少时，手动按需注册更适合新手理解和答辩。

- 误区三：路由懒加载会改变页面功能。
- 真实做法：路由懒加载只改变页面代码下载时机，不改变页面功能。

## 9. 老师可能会怎么问

### 问：后台包体偏大是什么意思？

可以回答：构建后的某个 JavaScript 文件太大，浏览器首次访问后台时需要下载更多代码。它不是运行错误，但会影响首次加载速度。

### 问：你是怎么优化的？

可以回答：我做了三层优化：Element Plus 由整包引入改成按需注册；后台路由由同步加载改成懒加载；Vite 构建时把 Vue、Element Plus 和其他依赖拆成不同 chunk。

### 问：为什么不直接用自动按需导入插件？

可以回答：当前后台组件数量有限，手动按需注册更直观，也不增加新依赖。这个项目面向课程答辩，优先保证我能讲清楚实现原理。

### 问：优化后功能有没有变化？

可以回答：没有变化。内容管理、分类、标签、来源、弹窗、提示框、确认框、深色模式都保持原有行为，只是加载结构变了。

## 10. 本阶段验收结果

- 后台 `npm run build` 通过
- 后台构建不再出现 500KB chunk 警告
- 前台 `npm run build` 通过
- 后端 `mvn -q -DskipTests compile` 通过
- 后台仍保留 Element Plus 的函数式消息提示和确认框
- 后台浅色 / 深色模式不受影响

## 11. 后续如何扩展

如果后续后台页面继续变多，可以继续做：

- 更细粒度的页面模块拆分
- 只在使用日期选择器的页面加载日期相关代码
- 对后台图标做按需加载
- 使用可视化包分析工具检查依赖来源
- 如果组件继续增多，再考虑自动按需导入插件
