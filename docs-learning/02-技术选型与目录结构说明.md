# 技术选型与目录结构说明

## 1. 为什么前端拆成两个 Vue 工程

你已经选择了“前后台两个前端工程”的方案。

这样做的好处：

- 前台门户和后台管理的视觉风格差异可以彻底拉开
- 老师看目录时，会明显感觉这是一个更完整的系统
- 后续部署和迭代时，前后台可以独立演进

这样做的代价：

- 目录会比单工程更复杂
- 两边都要维护依赖与启动命令

为什么仍然采用这个方案：

- 你希望这个项目长期发展，不只是课程一次性交作业
- 前台门户承担“产品形象”，后台承担“运营工具”，拆开更合理

## 2. 为什么后端用 Spring Boot + MyBatis-Plus

### Spring Boot

优点：

- 分层结构清楚
- 老师熟悉，答辩容易讲
- Controller / Service / Mapper 分工清晰

### MyBatis-Plus

优点：

- 比原生 MyBatis 少写很多样板代码
- 比 JPA 更容易理解“表和 SQL 到底怎么对应”
- 对课程项目和新手来说，更容易讲清楚

## 3. 为什么数据库用 MySQL 8

- 课程环境最常见
- 资料最多
- 和 Spring Boot 整合成熟
- 后续要做内容、标签、来源、检索时都够用

## 4. 为什么第一版不用复杂权限系统

第一版的重点不是“谁来登录”，而是“内容平台主流程是否完整”。

如果一开始做多角色权限，会出现两个问题：

- 时间被权限系统吃掉
- 内容平台主干反而做不完整

所以第一版采用：

- 无登录
- 单管理员演示模式

这并不代表架构低级，而是说明你做了阶段性取舍。

## 5. 当前目录如何理解

```text
AI前沿情报站/
├─ backend/           # 后端 API、业务逻辑、数据库访问
├─ frontend-portal/   # 前台内容展示门户
├─ frontend-admin/    # 后台内容管理端
├─ database/          # 建表脚本与后续初始化数据
└─ docs-learning/     # 教学文档、答辩文档、阶段记录
```

## 6. 代码内部命名为什么不用中文

虽然项目对外名称使用中文，但代码内部仍然要使用英文或拼音风格的 ASCII 标识。

原因：

- Java 包名不适合使用中文
- Maven artifactId 不适合使用中文
- Node 包名与部分构建工具对中文兼容性不稳定

所以我们采用：

- 对外展示：`AI前沿情报站`
- 代码内部：`aifrontier`

这在答辩时也是一个可讲的工程规范点。
