# 阶段 24：MiMo 申请增强包记录

## 1. 本阶段目标

本阶段目标不是继续扩展业务功能，而是把已经公开发布的项目整理成更适合 GitHub 展示和 MiMo Orbit 申请的形态：

- 提供真实运行截图，证明项目不是纯文字描述。
- 增强 README，让评审快速理解项目价值和完成度。
- 补充短版申请文案，方便直接填写申请表。
- 做基础敏感信息检查，避免开源材料泄露 API Key。

## 4. 为什么要做截图和短版材料

MiMo 申请不只是看代码是否存在，也会看项目是否有清晰目标、真实进展和可持续扩展方向。截图可以降低评审理解成本，短版文案可以把项目价值压缩成申请表能快速阅读的表达。

对这个项目来说，最应该强调的是：

- 它不是普通 CRUD，而是 AI 信息聚合平台。
- 它已经有完整前后端结构，不只是静态页面。
- 它已经接入 GitHub API 和百炼 AI 总结工作流。
- 它需要大模型 Token 的场景是真实的：来源整理、摘要生成、标签建议、推荐理由、周报生成。
- 它后续可以把 MiMo 作为新的 AI Provider 接进现有 AI 总结层。

## 5. 关键代码和文件怎么读

- `README.md`：面向 GitHub 访客和 MiMo 评审的总入口。
- `docs-learning/26-MiMo申请材料.md`：可以直接复制到申请表的中文材料。
- `frontend-admin/vite.config.js`：后台开发代理配置，本阶段修复了 `/api-settings` 路由冲突。
- `docs-learning/assets/mimo/`：申请截图资产目录。

## 6. 答辩时可以怎么讲

老师如果问“为什么 README 里放这么多截图”，可以这样回答：

> 因为这个项目既是课程作业，也是一个准备开源维护的 AI Builder 项目。截图能让别人快速看到系统已经有前台、后台、详情页、Dashboard 和 API 设置页，而不是只看代码猜效果。

老师如果问“为什么修 Vite 代理配置”，可以这样回答：

> 后台接口统一是 `/api/v1`，但之前 Vite 代理写成 `/api`，导致 `/api-settings` 这种前端路由被误判为后端接口。我把代理范围收窄到 `/api/v1`，这样前端路由和后端接口边界更清晰。

老师如果问“MiMo Token 准备用在哪里”，可以这样回答：

> 我不会把模型当成唯一数据源，而是把它作为 AI 总结层。GitHub、论文、官方博客和社区内容负责提供原始来源，MiMo 负责摘要、分类、打标签、生成推荐理由和周报。

## 7. 当前遗留问题

- 目前还没有在线部署链接，GitHub 展示依赖 README 和本地运行说明。
- 目前截图没有自动化刷新脚本，后续如果 UI 大改，需要重新截图。
- 当前 API 设置页没有登录鉴权，作为课程演示可以接受，生产环境需要管理员登录和密钥加密存储。

## 8. 下一阶段建议

下一阶段建议优先做“MiMo API Provider 适配设计与最小实现”，原因是当前 README 和申请材料已经准备好，继续把 MiMo 接入现有 AI 总结层，会让申请材料从“计划使用”进一步变成“已经具备适配入口”。

如果暂时不接 MiMo，也可以选择“GitHub 项目热度同步任务”，让平台从手动导入进一步走向定时数据更新。
