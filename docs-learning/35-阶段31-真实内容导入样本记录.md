## 4. 发现并修复的问题

真实数据比示例数据更容易触发数据库字段边界。

本次发现两个阻塞导入的问题：

- arXiv 论文摘要较长，直接写入 `ai_content.summary` 会超过 500 字
- arXiv 论文作者较多，直接写入 `ai_content.author_name` 会超过 100 字

修复方式：

- 后端 `ContentServiceImpl` 在写入内容主表前统一限制 `summary` 为 500 字以内
- 后端 `ContentServiceImpl` 在写入内容主表前统一限制 `author_name` 为 100 字以内
- 正文 `body_markdown` 和扩展字段 `extra_json` 仍保留完整摘要、作者和外部来源信息

这个修复不是为了某一条样本硬编码，而是为后续真实论文、社区内容和外部来源导入提供通用兜底。

不同导入方式的观察重点：

- AI 来源整理：看 AI 导读、推荐理由、来源说明和外部来源链接
- GitHub 项目导入：看仓库名、Star、语言、Topics、`github_repo` 外部引用
- arXiv 论文导入：看论文 ID、作者、PDF 链接、`arxiv_paper` 外部引用
- HuggingFace 论文导入：看 HF Paper 链接、社区热度、`huggingface_paper` 外部引用
- 普通新建内容：看它没有外部引用，说明手动录入和外部导入的差异

## 6. 个人使用时的观察重点

这组样本最有价值的地方，不是数量，而是它们覆盖了不同来源进入统一内容模型的路径：

- GitHub 仓库、arXiv 论文、HuggingFace Paper 都会转成统一的内容模型，并同时写入外部引用表。
- AI 来源整理会把真实来源交给模型生成导读，再由管理员保存为草稿。
- 普通新建内容用于对比人工录入路径和外部导入路径。
- 全部先保存为草稿，方便在发布前检查标题、摘要、标签、来源和正文。
