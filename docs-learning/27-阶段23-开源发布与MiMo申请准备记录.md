# 阶段23：开源发布与 MiMo 申请准备记录

## 1. 本阶段目标

本阶段把项目从“课程大作业项目”整理成“可公开发布的开源 AI Builder 项目”，并准备 Xiaomi MiMo Orbit 百万亿 Token 申请材料。

主要目标：

- 重启后端，让后台 API 设置页可用。
- 增加开源安全配置，避免提交密钥和构建产物。
- 补充 MIT License。
- 重写 README，让项目更适合 GitHub 展示。
- 准备 MiMo 中文申请材料。
- 初始化 Git 仓库并发布到 GitHub public 仓库。

## 2. 本阶段涉及的技术点

- Git 初始化与 GitHub 发布
- `.gitignore` 开源安全配置
- 环境变量示例文件 `.env.example`
- README 开源项目包装
- API Key 安全说明
- MiMo 申请材料整理

## 3. 为什么要做开源安全整理

项目中已经接入百炼 API 和 GitHub API。真实 API Key 不能写入代码，也不能提交到 GitHub。

本阶段采用的策略：

- `.env` 文件不提交。
- README 和 `.env.example` 只写变量名，不写真实密钥。
- 后台 API 设置页中的密钥只保存在后端运行内存。
- `application.yml` 使用环境变量读取数据库密码、百炼 Key 和 GitHub Token。
- `node_modules`、`dist`、`target` 等构建产物不提交。

## 4. 为什么 README 要重写

原 README 更像课程项目进度说明，不太适合开源展示和 MiMo 申请。

新的 README 强调：

- 项目定位：AI 信息聚合与精选平台。
- 技术架构：Vue + Spring Boot + MySQL。
- AI 能力：百炼总结层和 GitHub API。
- 安全设计：运行时 API 设置和 Markdown 清洗。
- 后续路线：GitHub 同步、MiMo Provider、AI 周报。

## 5. 老师或评审可能会怎么问

**问：这个项目为什么适合申请 Token？**

答：它已经具备信息来源、内容管理、AI 总结和前台展示的完整链路。大模型 Token 可以用于批量总结 GitHub README、论文摘要、官方动态和社区实践，不是为了简单聊天，而是作为内容平台的生产能力。

**问：为什么开源仓库里没有真实 API Key？**

答：真实 API Key 属于敏感信息，不能提交到开源仓库。项目通过后台 API 设置页和环境变量支持配置，既能演示，也能保护密钥。

**问：为什么 API Key 不存数据库？**

答：当前项目还没有登录和权限系统。第一版把密钥保存在后端内存中，重启后失效，更适合课程演示。后续可以扩展为管理员权限加加密落库。

## 6. 后续如何扩展

发布完成后，建议优先做两件事之一：

1. GitHub 项目热度同步任务：让已有 GitHub 外部引用能刷新 Star、语言、更新时间。
2. MiMo API Provider：把 MiMo 作为新的 AI 总结模型接入现有 AI 总结层。

如果目标是提高 MiMo 申请通过率，建议优先做 MiMo API Provider 适配计划和截图材料。
