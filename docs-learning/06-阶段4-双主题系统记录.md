# 阶段4：双主题系统记录

## 1. 本阶段目标

这一阶段的目标，是让门户端同时具备浅色模式和深色模式，并且做成可维护的主题系统，而不是简单换几个背景色。

## 2. 为什么现在做双主题

原因有三个：

- 门户已经进入高保真设计阶段，适合开始补齐视觉系统能力
- 双主题能直接体现你有设计 token 和产品体验意识
- 现在先把主题架构搭好，后续继续加页面时就不用每个页面重写一遍深色逻辑

## 3. 本阶段做了什么

### 3.1 新增主题状态管理

新增文件：

- `frontend-portal/src/stores/useThemeStore.js`

这个 store 负责：

- 保存当前主题
- 处理主题初始化
- 处理手动切换
- 持久化用户选择

### 3.2 让主题在应用启动时初始化

修改文件：

- `frontend-portal/src/main.js`

这样做的目的：

- 在应用挂载前先确定主题
- 避免页面先按默认模式闪一下，再切到另一种模式

### 3.3 把切换入口放到门户顶栏

修改文件：

- `frontend-portal/src/components/PortalTopbar.vue`

为什么放在这里：

- 顶栏是所有门户页面共用的组件
- 入口统一，用户最好找
- 后续新页面也不用重复加切换按钮

### 3.4 把颜色体系改造成真正的双主题 token

修改文件：

- `frontend-portal/src/styles/global.css`

核心做法：

- 浅色模式放在 `:root`
- 深色模式放在 `:root[data-theme='dark']`

通过 token 统一控制：

- 页面背景
- 卡片表面
- 边框
- 文本层级
- 输入框
- 次按钮
- 主按钮渐变
- 阴影
- 代码块颜色

## 4. 默认策略怎么设计

当前规则：

- 如果用户没选过主题，第一次进入时跟随系统主题
- 如果用户手动切换过，就优先记住用户选择

这是一种比较合理的策略，因为它兼顾了：

- 初次体验
- 个性化偏好
- 刷新后的稳定性

## 5. 为什么这比“做个深色样式”更专业

因为真正的主题系统，不是每个页面单独写一套深色样式，而是：

- 全局状态统一管理
- 页面结构不变
- 视觉通过 token 切换

这样后续维护成本更低，也更容易讲清楚。

## 6. 这一阶段适合答辩时怎么讲

### 问：为什么主题要做成 store

答：

因为主题属于全局状态，不应该交给某个单独页面管理。
放在 store 之后，所有页面和组件都可以共享这一份状态。

### 问：为什么用 CSS 变量做双主题

答：

因为 CSS 变量适合管理整套设计 token。
新增页面时只要继续复用这些 token，就能自动兼容浅色和深色。

### 问：为什么不直接做两套页面

答：

因为那样维护成本高，而且结构会分裂。
更合理的做法是保留同一套页面结构，用主题层控制视觉差异。

## 7. 这一阶段体现了什么工程意识

这一阶段体现的重点不是“多了一个按钮”，而是：

- 全局状态管理意识
- 设计 token 意识
- 组件复用意识
- 可维护性意识

## 8. 当前边界

这轮双主题目前只先落在门户端。

也就是说：

- 前台门户已支持浅色 / 深色模式
- 后台管理端暂时还是单主题

这是合理的阶段性取舍，因为当前最强调产品感和品牌感的是门户端。
