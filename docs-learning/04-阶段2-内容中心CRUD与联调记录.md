# 阶段2：内容中心 CRUD 与前后台联调记录

## 1. 本阶段目标

这一阶段的任务，是把项目从“只有工程骨架”推进成“有真实业务流”的系统。

具体来说，要完成三件事：

- 后端真正连上数据库
- 内容、分类、标签、来源四块具备真实 CRUD
- 前台和后台开始读取真实接口，不再只是静态占位

## 2. 本阶段做了什么

### 2.1 后端从骨架变成真实业务服务

本阶段新增了这些核心层次：

- `entity`
- `mapper`
- `service`
- `service/impl`
- `dto/request`
- `vo`
- `controller/admin`
- `controller/portal`

这一步非常重要，因为它意味着后端已经不是“只有一个健康接口”的演示程序，而是具备真实业务结构的系统。

### 2.2 已实现的后端接口

后台接口：

- 分类管理
- 标签管理
- 来源管理
- 内容管理
- 内容状态切换
- 内容表单选项加载

前台接口：

- 首页精选内容
- 内容列表
- 内容详情

## 3. 为什么要这样分层

### Controller

作用：

- 接收前端请求
- 接收参数
- 返回统一结构

它不应该直接写太多业务逻辑。

### Service

作用：

- 写业务规则
- 校验关联关系
- 处理标签绑定
- 处理发布状态

这是老师很容易问到的一层。

### Mapper

作用：

- 直接和数据库交互
- 负责表级数据访问

为什么这样设计好讲：

- Controller 负责接口层
- Service 负责业务层
- Mapper 负责数据访问层

这就是标准的三层结构。

## 4. 为什么内容模块是系统核心

本项目不是普通管理系统，所以不是“用户表”或“订单表”当核心。

这里真正的核心是：

- `ai_content`

它承载所有要展示给前台的 AI 内容。

但只靠一张内容表不够，所以还要配套：

- `ai_category`
- `ai_tag`
- `ai_source`
- `ai_content_tag`
- `ai_content_external_ref`

这样才能既支持当前版本，又支持以后扩展。

## 5. 为什么要独立出来源表

因为这个项目后续要接：

- GitHub
- arXiv
- 官方博客
- 科技社区
- 人工录入

如果来源直接写死在内容表的一两个字符串字段里，后面就很难做：

- 来源管理
- 来源筛选
- 来源统计
- 外部同步

所以来源必须独立成表。

## 6. 为什么内容和标签要多对多

因为一条内容可能同时属于多个语义维度。

比如：

- 一条内容既是 GitHub 项目
- 又和 Agent 有关
- 还涉及工程优化

如果把标签写成单字段，就不利于：

- 前台筛选
- 标签聚合
- 后续推荐

所以这里用：

- `ai_content_tag`

来做多对多关系。

## 7. 本阶段修复过的真实问题

### 问题 1：内容列表空关键字时报错

原因：

- 在构造查询条件时，空关键字仍然直接调用了 `trim()`

修复方式：

- 先做空值判断，再决定是否参与查询

这个问题很适合答辩时讲，因为它体现了：

- 不是接口能跑就结束
- 还要考虑边界条件

### 问题 2：分页能查到数据，但总数一直是 0

原因：

- MyBatis-Plus 没有配置分页拦截器

修复方式：

- 新增 `MybatisPlusConfig`
- 注册 `PaginationInnerInterceptor`

这个点也很适合答辩讲：

> 框架不是“引入依赖就自动完整生效”，分页这类能力还需要显式配置。

## 8. 当前前台和后台分别已经能做什么

### 前台门户

- 首页读取精选内容和最新内容
- 内容广场读取已发布内容
- 详情页读取真实正文

### 后台管理端

- 来源管理支持增删改查

## 10. 老师可能会怎么问

### 问：为什么不用一张表直接存分类名称和标签字符串

答：

因为分类、标签、来源都属于可复用的组织维度。
独立成表后，才能支持后台维护、前台筛选、统计分析和后续推荐。

### 问：为什么内容详情要单独接口，不直接在列表里把所有字段都返回

答：

因为列表页和详情页关注的数据粒度不同。
列表页只需要摘要型字段，详情页需要正文、标签、来源等完整信息。
拆分接口更合理，也更符合真实系统设计。

### 问：为什么后台要有内容选项接口

答：

因为内容表单依赖分类、标签、来源、内容类型、发布状态这些选项数据。
单独做一个选项接口，可以让前端一次性拉取表单依赖，表单逻辑更清晰。
