# 开发踩坑记录

本文档记录开发过程中遇到的所有问题和解决方案，防止重复犯错。

## 1. Vue + Element Plus 按需导入陷阱

### v-loading 指令未注册
- **问题**：`v-loading="loading"` 导致整个组件渲染失败，所有数据更新都不生效
- **原因**：项目使用 Element Plus 按需导入（tree-shaking），`v-loading` 指令没有全局注册
- **解决**：不要用 `v-loading`，改用 `v-show`/`v-if` 控制加载状态
- **教训**：Element Plus 按需导入时，指令（v-loading, v-infinite-scroll 等）不会自动注册

### el-input vs native textarea
- **问题**：native `<textarea>` 配合 Vue ref 更新可能有响应式问题
- **解决**：统一使用 `<el-input type="textarea">`

## 2. Axios 拦截器导致的数据层级问题

### response.data 已被拦截器剥离
- **问题**：代码写 `res.data.data` 但实际只需要 `res.data`
- **原因**：http.js 拦截器已经做了 `response.data`，返回的 `res` 就是 API 响应体 `{code, message, data}`
- **正确**：`const data = res.data` 就是业务数据
- **错误**：`const data = res.data.data` 多取了一层
- **教训**：写前端 API 调用前先看 http.js 拦截器是否做了转换

## 3. MyBatis-Plus 字段命名映射

### snake_case vs camelCase
- **问题**：数据库列名 `star_growth7d` 和实体字段 `starGrowth7d` 映射失败
- **原因**：MyBatis-Plus 默认驼峰映射，但列名必须是标准的 `star_growth_7d` 格式
- **解决**：列名用 `star_growth7d`（不加第二个下划线）或实体字段用 `@TableName` 注解指定
- **教训**：创建数据库列名时，确保和 Java 实体的驼峰命名能正确映射

## 4. API Key 加密持久化

### master key 不匹配
- **问题**：重启后 API Key 显示未启用
- **原因**：加密时用的 `API_MASTER_KEY` 和解密时不一致（可能是不同的启动命令）
- **解决**：确保每次启动都传相同的 `API_MASTER_KEY` 环境变量
- **教训**：API Key 一旦保存，master key 不能变

## 5. GitHub API 限速

### 无 Token 限速 60 次/小时
- **问题**：GitHub 采集返回 0 条数据
- **原因**：没有配置 GitHub Token，API 限速已用完
- **解决**：去后台 API 设置页面填入 GitHub Personal Access Token
- **教训**：GitHub API 无 token 只有 60 次/小时，有 token 5000 次/小时

## 6. 后端进程管理

### 端口冲突
- **问题**：重启后端时报 "Port 8080 was already in use"
- **原因**：旧进程没被完全杀死
- **解决**：先 `lsof -i :8080 | awk 'NR>1{print $2}' | xargs kill -9` 再启动

### Mac 睡眠导致 HikariPool 警告
- **问题**：`Thread starvation or clock leap detected`
- **原因**：Mac 合盖睡眠后时钟跳变
- **影响**：不影响功能，可以忽略

## 7. 前端字段命名一致性

### API 返回 camelCase，模板用 snake_case
- **问题**：`selectedCandidate.published_at` 取不到值
- **原因**：API 返回 `publishedAt`（camelCase）
- **解决**：前端所有字段统一用 camelCase
- **教训**：Java 后端 + MyBatis-Plus 默认返回 camelCase，前端不要用 snake_case

## 8. 种子数据不能编造

### 用户明确拒绝假数据
- **问题**：插入了编造的种子数据（如 "Gemini 2.5 Pro 登顶"）
- **用户反馈**："你这些文章都是编的吧"
- **解决**：只用真实 API 采集真实数据
- **教训**：永远不要编造数据，宁可显示"暂无数据"也不要造假

## 9. 候选审核页面设计

### 不需要"全部"Tab
- **原因**：各模块数据格式不同，混在一起没有意义
- **解决**：默认显示第一个有数据的分类

### 详情应该在弹窗中，不是页面底部
- **原因**：底部面板占用太多空间，体验差
- **解决**：用 el-dialog 弹窗展示详情

## 10. TrendRadar 架构原则

### 不要按 total stars 排序
- **问题**：transformers/llama 永远霸榜
- **解决**：用 Trend Score 算法（star 增长 + 时间衰减 + AI 相关性）

### 数据管道分层
```
采集 → 候选表(原始数据) → 管理员审核/AI处理 → Content表(发布)
```
- 每一步都可编辑
- 管理员能看到原始数据和 AI 处理结果

### AI 不只是摘要
- 需要：中文摘要、技术标签、方向分类、难度评级、适合人群、学习路线
- 提示词要可编辑，让管理员控制 AI 输出格式
