# 阶段 19：前台内容发现页搜索体验增强记录

## 1. 本阶段目标

本阶段增强前台内容广场的搜索与筛选体验，让它更像真实的信息发现产品。

本阶段不新增数据库表，不引入全文搜索引擎，不接 Elasticsearch，也不做实时联想搜索。重点是在现有 MySQL 查询能力上补齐排序、标签筛选、URL 状态保持和更清晰的结果反馈。

## 2. 为什么现在做这个阶段

后台已经具备内容管理、GitHub 手动导入、外部引用、Markdown 安全清洗和 Dashboard 统计。接下来需要回到用户侧，让前台内容发现页更好用。

内容聚合平台的核心体验不是只有详情页，而是用户能不能快速找到内容：

- 按关键词找内容
- 按内容类型筛选
- 按分类、来源和标签缩小范围
- 按最新、热门、精选排序
- 刷新页面后保持检索状态

这些能力完成后，前台会更像真正的 AI 信息聚合平台。

## 3. 本阶段涉及的技术点

- Spring Boot 请求参数扩展
- MyBatis-Plus 动态排序
- 内容标签筛选
- Vue Router 查询参数同步
- 前台筛选状态恢复
- 结果反馈与空状态设计
- 不引入搜索引擎时的轻量检索方案

## 5. 后端新增了什么

门户内容列表接口新增 `sortBy` 参数：

```text
GET /api/v1/portal/contents?sortBy=popular
```

支持三种排序：

```text
latest    最新发布
popular   最多浏览
featured  精选优先
```

无效排序会按 `latest` 处理，不会报错。这样即使用户手动改 URL，也不会让页面崩掉。

首页接口 `GET /api/v1/portal/home` 现在也会返回标签列表，前台可以展示热门标签快捷筛选。

## 6. 为什么要做 URL 查询参数同步

以前筛选状态只存在页面内存里，刷新页面就丢失。现在关键词、类型、分类、来源、标签、排序和页码都会同步到 URL。

例如：

```text
/contents?keyword=GitHub&contentType=project&sortBy=popular&pageNum=1
```

这样有三个好处：

- 刷新页面后筛选状态保持
- 可以把某个检索结果链接发给别人
- 答辩时可以直接演示“状态和路由同步”

## 7. 标签筛选怎么实现

后端本来已经有 `tagId` 参数，但前台没有使用。本阶段把标签快捷筛选接入内容广场。

实现方式是：

1. 前台从首页接口读取标签列表。
2. 内容广场展示前 8 个标签。
3. 点击标签后把 `tagId` 写入 URL。
4. 后端根据 `ai_content_tag` 关联表筛选内容。

这说明标签表不是摆设，而是真正参与内容发现。

## 8. 老师可能会怎么问

### 问：为什么不直接上 Elasticsearch？

可以回答：当前项目数据量小，课程第一版优先保证结构清晰和可运行。MySQL 的关键词、筛选和排序已经足够支撑第一版。后续数据量变大或需要全文检索时，再引入搜索引擎。

### 问：为什么要把筛选条件放到 URL？

可以回答：这样刷新页面不会丢失状态，也方便分享检索结果。它体现了前端路由状态管理，而不是把所有状态都藏在组件内存里。

### 问：为什么标签筛选要通过关联表？

可以回答：一条内容可以有多个标签，一个标签也可以对应多条内容，这是多对多关系，所以通过 `ai_content_tag` 关联表实现。

### 问：排序为什么只做三种？

可以回答：第一版先做最常用的三种：最新、热门、精选。它们分别对应发布时间、浏览量和精选级别，已经能覆盖内容平台的主要浏览方式。
