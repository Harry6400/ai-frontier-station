# 阶段3：门户设计强化记录

## 1. 本阶段目标

这一阶段的重点，不是继续堆后台功能，而是把前台真正做成一个“像产品”的内容门户。

要解决的问题有三个：

- 首页为什么看起来像一套平台而不是课程作业页面
- 列表页为什么像内容发现页而不是普通表格替代物
- 详情页为什么能体现“内容平台”的阅读体验

## 2. 本阶段采用的设计思路

这一步我按你要求，优先参考了 `huashu-design` 的关键原则：

- 系统优先，不靠堆 filler 内容填页面
- 避免 AI slop，不走廉价紫渐变和通用 SaaS 套路
- 先建立视觉系统，再改页面
- 前台要承担品牌感和内容编排能力

所以这次不是单纯换背景，而是先做了这些基础层：

- 统一色彩与表面层级
- 统一字体与排版节奏
- 统一卡片、按钮、标签、导航规范
- 统一内容类型和时间格式化方式

## 3. 这一阶段具体做了什么

### 3.1 建立了共享门户组件

新增：

- `frontend-portal/src/components/PortalTopbar.vue`
- `frontend-portal/src/components/ContentCard.vue`
- `frontend-portal/src/utils/content.js`

为什么要这样做：

- 顶部导航在所有页面都复用
- 内容卡片在首页和列表页都复用
- 内容类型、日期格式化不应该每个页面重复写

这就是“设计系统”和“代码复用”结合的体现。

### 3.2 重做了门户首页

首页不再只是“标题 + 三个卡片”。

现在它包含：

- 品牌导航区
- 英雄区（Hero）
- 实时情报区
- 精选卡片区
- 栏目地图区
- 平台结构说明区

这样做的意义：

- 首页一打开就能看出项目方向
- 老师能一眼看到这不是普通后台附带前台
- 也能体现未来扩展成平台的潜力

### 3.3 重做了内容广场

列表页现在强调“内容发现”而不是“内容罗列”。

做了这些改动：

- 筛选区和页面说明分区清楚
- 结果区有独立标题和分页状态
- 内容采用统一内容卡片样式
- 列表页整体更像内容产品而不是后台数据面板

### 3.4 重做了内容详情页

详情页是这一阶段非常重要的地方，因为它最能体现内容平台的“阅读体验”。

现在详情页包含：

- 内容头图式布局
- 文章摘要与元信息
- 标签区
- 原始链接入口
- 侧边内容画像
- Markdown 正文阅读区

### 3.5 引入 Markdown 渲染

门户端新增了 `marked` 依赖，用于把正文 Markdown 渲染成真正的文章结构。

为什么这一步重要：

- 详情页终于不像调试页了
- 也和后端“正文先用 Markdown 存储”的设计形成闭环
- 以后接论文摘要、项目说明、技术实践时更自然

## 4. 为什么这一步很适合答辩

因为这一步不仅是“好看一些”，而是能讲出完整设计逻辑。

### 可以讲的点 1：前台为什么要有设计系统

答：

因为内容平台的前台不是随便放数据，而是要承担品牌形象、内容组织和阅读体验。
如果没有统一导航、卡片、按钮、标签和排版规范，页面很快就会变乱。

### 可以讲的点 2：为什么要抽共享组件

答：

因为首页和列表页都在使用内容卡片，多个页面都在使用顶部导航和内容格式化逻辑。
如果不抽组件，后续页面一多就很难维护。

### 可以讲的点 3：为什么详情页要用 Markdown 渲染

答：

因为正文不只是数据库里的一段文本，而是内容平台最核心的展示对象。
Markdown 既方便当前阶段维护，也方便未来接入不同来源的内容。

## 5. 这一阶段改进了哪些“产品感”

### 首页的产品感

- 不再只是课程说明
- 开始像内容平台的门户首页
- 能同时展示内容、栏目和系统逻辑

### 列表页的产品感

- 更像内容发现页
- 筛选与结果区分清楚
- 卡片语言统一

### 详情页的产品感

- 正文有阅读结构
- 元信息更完整
- 侧边信息更有“编辑后台与前台内容对应”的感觉

## 6. 本阶段遇到的真实工程问题

### 问题 1：门户旧开发服务里残留了旧依赖报错

现象：

- 开发时旧的热更新缓存里还残留了之前 `element-plus` 的错误提示

处理方式：

- 确认当前代码已移除相关引用
- 重新启动门户开发服务

这个问题提醒你：

> 运行时问题不一定都是当前代码的逻辑错误，也可能是开发服务缓存状态造成的。

### 问题 2：新增 `marked` 后需要重新让开发服务加载依赖

处理方式：

- 安装依赖
- 重启前台开发服务
- 再次构建验证

## 7. 当前这一版为什么更像“AI 内容平台”

因为它已经开始具备下面这些特征：

- 首页有内容编排
- 列表页有内容发现
- 详情页有内容阅读
- 项目说明页能直接支撑答辩

这几个点合起来，比“几个管理页面 + 一个简单首页”更像真正的平台。

## 8. 老师可能会怎么问

### 问：为什么前台也要花这么多精力设计

答：

因为这个项目的目标不只是做管理，而是做“AI 内容聚合与精选平台”。
如果前台没有产品化设计，就很难体现这个项目的业务定位。

### 问：为什么要把内容卡片抽成组件

答：

因为内容卡片是平台里的高频展示单元。
抽成组件后，首页、列表页、后续专题页都可以复用，维护成本更低。

### 问：为什么不用更复杂的富文本编辑器

答：

第一版更重要的是把内容主链路跑通并讲清楚。
Markdown 对当前阶段更轻、更稳，也更利于展示和扩展。

## 9. 下一阶段建议

接下来更适合继续补的，不是再大改首页，而是：

- 继续增强前台筛选交互
- 增加前后台更多数据状态反馈
- 优化后台内容表单体验
- 准备下一步的外部来源接入设计
