# 阶段 12：外部引用关系展示记录

## 1. 本阶段目标

上一阶段已经让内容支持 `extra_json` 扩展字段和 `sourceType` 来源类型。本阶段继续向真正的 AI 信息聚合平台靠近：把数据库中已经预留的 `ai_content_external_ref` 外部引用表接入后端和页面展示。

本阶段仍然不做自动采集，而是先完成：

- 后端能读取外部引用表
- 内容详情接口能返回外部引用列表
- 前台详情页能展示外部引用关系
- 后台内容编辑预览能看到外部引用记录
- 文档能讲清楚“内容表”和“外部系统记录”之间的关系

## 4. 为什么这样设计

一个真正的信息聚合平台，不能只保存“整理后的文章”，还要保存“它来自外部哪个原始对象”。

例如：

- GitHub 项目内容对应一个 GitHub 仓库
- 论文速递内容对应一篇 arXiv 论文
- 公司动态内容对应一篇官方博客文章
- 技术实践内容对应一篇社区文章或经验帖

所以系统需要两层数据：

- `ai_content`：平台内部统一内容，负责展示、分类、标签、发布状态
- `ai_content_external_ref`：外部系统引用，负责记录外部 ID、外部链接、原始数据快照、同步时间

这样做的好处是：前台和后台使用统一内容模型，而未来同步模块可以通过外部引用表追踪原始数据。

## 5. 为什么不直接把外部 ID 放进内容表

如果只接一种数据源，把外部 ID 放到内容表也能工作。

但这个项目未来要支持：

- GitHub
- 论文站点
- 官方博客
- 技术社区
- 更多内容源

不同来源的外部 ID 含义不同，一个内容未来甚至可能关联多个外部引用。所以单独建 `ai_content_external_ref` 更清晰。

答辩时可以这样说：

> 内容表保存平台内部要展示的统一内容，外部引用表保存内容和外部系统之间的映射关系。这样以后做同步、去重、更新追踪时，不需要改内容主表结构。

## 6. 关键代码应该怎么读

### 6.1 Entity

`ContentExternalRef.java` 对应数据库表 `ai_content_external_ref`。

主要字段：

- `contentId`：关联平台内容
- `refType`：外部引用类型，例如 `github_repo`
- `externalId`：外部系统中的 ID
- `externalUrl`：外部原始链接
- `rawPayloadJson`：原始数据快照
- `syncedAt`：同步时间

### 6.2 Mapper

`ContentExternalRefMapper.java` 继承 MyBatis-Plus 的 `BaseMapper`。

这说明当前不需要手写 SQL，也能进行基础查询。

### 6.3 Service

`ContentServiceImpl.java` 新增了 `findExternalRefs`。

它会按 `contentId` 查询外部引用，并按同步时间和 ID 倒序排列。

内容详情组装时会设置：

```java
vo.setExternalRefs(findExternalRefs(content.getId()));
```

### 6.4 VO

`ContentDetailVO.java` 新增：

```java
private List<ContentExternalRefVO> externalRefs;
```

这表示前台详情页和后台编辑详情都能拿到外部引用列表。

## 7. 前台怎么展示

前台详情页在右侧信息区新增“外部引用记录”。

它会显示：

- 外部引用类型
- 外部 ID
- 外部引用说明
- 最近同步时间
- 可点击的外部链接

这样老师能直观看到：这条内容不是孤立文章，而是可以和外部系统记录关联。

## 8. 后台怎么展示

后台内容编辑弹窗右侧预览区新增“外部引用记录”。

这对后台管理员有两个价值：

- 编辑内容时能知道这条内容是否来自外部系统
- 后续接自动采集时，可以在这里检查同步来源和外部 ID

## 9. 老师可能会怎么问

### 问：`extra_json` 和 `external_ref` 有什么区别？

可以回答：

> `extra_json` 是内容本身的扩展字段，例如语言、Star 趋势、论文来源；`external_ref` 是内容和外部系统记录的关联，例如 GitHub 仓库 ID、arXiv ID、官方博客链接。一个描述内容属性，一个描述外部映射关系。

### 问：为什么需要 `raw_payload_json`？

可以回答：

> 后续自动采集时，第三方接口返回的数据可能很多，不一定都立刻展示。保存原始快照可以方便后续重新解析、排错和追踪同步结果。

### 问：现在没有自动采集，这个表有什么用？

可以回答：

> 当前先用初始化数据模拟外部引用关系，让系统结构完整。后续接 GitHub 或论文 API 时，只需要由同步服务写入这张表，前台和后台展示逻辑已经准备好了。
