# AI 前沿情报站 — 发展规划

> 更新时间：2026-05-17
> 本文档是项目从"课程大作业"转型为"个人 AI 信息雷达"的完整发展路线。
> 开发 agent 请先读 `AGENTS.md`，再读本文件。

---

## 一、项目现状

### 已完成的能力（阶段 1-32）

- Vue3 前台门户 + 后台管理 + Spring Boot 后端 + MySQL 全栈
- arXiv 论文搜索导入、HuggingFace Daily Papers 导入
- GitHub 仓库查询/搜索/导入 + 热度同步
- AI 总结层（百炼 DashScope + MiMo 双 Provider）
- 后台 AI 来源整理工作台（人工输入 → AI 生成导读 → 确认入库）
- JWT 登录、API Key AES-256-GCM 加密存储
- 浅色/深色主题、Markdown 安全清洗、Dashboard 统计

### 核心问题

目前所有数据导入都是**手动**的（后台点按钮搜索 → 选择 → 导入），还没有"自动定期采集 + AI 内化 + 展示"的闭环。

---

## 二、项目定位转型

从"人工运营的内容管理系统"变成"**为吴皓睿个人定制的 AI 信息雷达**"。

### 三个核心层次

1. **医学影像/CT 去噪论文层** — 最核心，服务于科研（跟毛学长做 low-dose CT 去噪方向）
2. **AI 前沿资讯层** — 更广的信息面，来源于全球多渠道（不用 X/Twitter API，因为付费）
3. **个人网站层** — 名片/歌单/账号等个人展示（从主站导航跳转）

### 内容原则

- **不是爬虫搬运**：所有外部信息都要经过 AI 内化总结后才上架
- **保留原文链接**：用户想看原信息可以直接跳转
- **追求真实权威**：来源必须是 arXiv、官方博客、GitHub 等可信渠道
- **个性化**：内容围绕 CT 去噪、医学影像、AI 前沿，不是通用新闻聚合

## 三、发展阶段规划

### 阶段 A：定时采集引擎（最高优先级）

**目标**：把"人去后台点一下"变成"系统自动跑"。

#### A1：arXiv 定时采集器

- 后端加定时任务模块（Spring `@Scheduled` 或 Quartz）
- 第一个采集器：arXiv API 定时拉取，关键词过滤
- 核心关键词（按你的研究方向）：
  - `low-dose CT`
  - `CT denoising`
  - `medical image denoising`
  - `diffusion model for CT`
  - `sparse-view CT`
  - `3D CT reconstruction`
  - `CT artifact reduction`
  - `noise reduction medical imaging`
- 频率：每天跑一次
- 新论文自动入库为"待审核"草稿状态
- 用 `arxiv_id` 做去重，避免重复入库

#### A2：HuggingFace Daily Papers 定时采集器

- 每天拉取 HuggingFace Daily Papers
- 过滤 `computer-vision` / `medical-imaging` 相关论文
- 用 `paper_id` 做去重

#### A3：arXiv 作者追踪

- 追踪领域内活跃作者的新论文
- 初期手动维护作者列表，后续可以扩展

#### A4：采集日志

- 加一张 `ai_fetch_log` 表
- 记录每次采集：时间、来源、抓到条数、成功/失败、失败原因
- 后台 Dashboard 可以展示采集状态

#### 技术要点

- 现有 `ArxivServiceImpl` 和 `HuggingFaceServiceImpl` 已经有 API 调用逻辑，只需要包一层定时调度 + 关键词过滤
- Spring Boot Starter 自带 `@Scheduled` 支持，加 `@EnableScheduling` 即可
- 防重复入库：arXiv 用 `arxiv_id`，HuggingFace 用 `paper_id`，GitHub 用 `full_name`

---

### 阶段 B：AI 内化管线

**目标**：新论文入库后自动触发 AI 总结，不是原文搬运。

#### B1：自动 AI 总结触发

- 新论文入库后，自动调用百炼/MiMo 总结（复用已有的 `source-summary` 接口逻辑）
- AI 生成内容：
  - **一句话核心贡献**（给首页卡片用）
  - **方法摘要**（3-5 句话，面向本科生背景水平）
  - **与 CT 去噪领域的关联度评分**（1-10）
  - **推荐理由**（为什么你该看这篇）
  - **关键图表描述**（如果有）
- 结果存入 `extra_json`，前台详情页直接展示

#### B2：人工审核队列

- AI 总结完 → 状态设为"待审核"
- 后台新增"待审核"筛选入口
- 你确认后才发布到前台
- 支持批量审核（全部发布 / 全部忽略）

#### B3：总结质量优化

- 针对论文类型优化 prompt：
  - 方法论文：重点总结创新点、网络结构、损失函数
  - 综述论文：重点总结涵盖范围、关键发现、趋势
  - 应用论文：重点总结应用场景、效果对比
- 支持重新生成（对不满意的总结手动重跑）

#### 成本注意

- 每天可能新增 10-30 篇论文
- 用 MiMo API 的话注意 token 消耗
- 可以设置"只对关联度评分 >= 6 的论文生成详细总结"来控制成本

---

### 阶段 C：AI 资讯多源接入

**目标**：不用 X/Twitter API（付费），通过其他渠道获取全球 AI 最新消息。

#### C1：RSS/Atom 源（最容易实现，优先做）

- OpenAI Blog: `https://openai.com/blog/rss.xml`
- Google AI Blog: `https://blog.google/technology/ai/rss/`
- Meta AI Blog
- DeepMind Blog
- HuggingFace Blog
- Papers With Code 最新 trending
- 实现一个通用的 RSS 解析器，后续加源只需要配置 URL

#### C2：GitHub Trending

- 每日/每周 AI 相关 trending repos
- 已有 GitHub API 集成，扩展筛选条件即可
- 按 `topic: machine-learning, deep-learning, computer-vision` 过滤
- 用 `full_name` 去重

#### C3：Hacker News / Reddit

- Hacker News API 公开免费
- Reddit r/MachineLearning 有公开 API
- 按 score 过滤热门帖子
- 只取标题 + 链接 + 简介，不做全文抓取

#### C4：中文源（优先级低，可选）

- 机器之心 RSS
- 量子位 RSS
- 知乎 AI 话题热榜（需要爬虫，复杂度高，建议后期再做）

#### 核心原则

每个源拉下来后都走 AI 总结管线，不是原文搬运。前台展示的是 AI 内化后的摘要 + 原文链接。

---

### 阶段 D：前台门户改造

**目标**：前台布局从"通用内容平台"改为"个人信息雷达"。

#### D1：首页分区

- **「今日论文速递」** — 最新医学影像/CT 去噪论文卡片，突出关联度评分
- **「AI 前沿动态」** — 全球 AI 资讯精选，按来源分组
- **「值得关注」** — AI 评出的本周最值得深读的内容
- **「最新入库」** — 所有新内容的时间线

#### D2：论文专区页

- 按子方向分类：
  - CT 去噪（Denoising）
  - CT 重建（Reconstruction）
  - 医学图像分割（Segmentation）
  - 扩散模型应用（Diffusion Models）
  - 其他医学影像
- 按时间线浏览
- 每篇都有 AI 生成的"速读版"
- 支持按关联度评分排序

#### D3：资讯流页

- 类似信息流，按时间倒序
- 支持按来源/标签/重要性筛选
- 卡片设计区分"论文"和"资讯"两种类型

#### D4：搜索增强

- 全文搜索（论文标题、摘要、AI 总结内容）
- 支持按日期范围筛选
- 搜索结果高亮关键词

---

### 阶段 E：个人网站模块

**目标**：一个独立的个人展示页面，从主站导航栏跳转。

#### E1：个人主页

- 路由：`/me` 或 `/about`
- 内容：
  - 个人简介（姓名、学校、专业、年级）
  - 研究方向（医学影像、CT 去噪）
  - 联系方式（邮箱、GitHub）
  - 技能标签

#### E2：社交账号聚合

- GitHub、B站、知乎等平台的账号名和链接
- 不嵌入内容，只做链接跳转

#### E3：歌单

- 嵌入网易云音乐或 Spotify 播放器
- 或者手动列出歌单 + 外链

#### E4：个人名片风格

- 卡片式设计，可分享
- 支持深色/浅色主题（复用现有主题系统）

---

### 阶段 F：部署上线

**目标**：本地跑没意义，需要一个随时能打开的网站。

#### 方案一：云托管（推荐）

- 前端：Cloudflare Pages 或 Vercel（免费、快）
- 后端：Railway 或 Render（有免费 tier 支持 Java）
- 数据库：PlanetScale 或 Railway MySQL

#### 方案二：AutoDL 自托管

- 在你的 AutoDL 实例上 Docker 部署全套
- 用 Nginx 反向代理
- 需要注意实例重启后数据持久化

#### 通用要求

- 配置固定 JWT_SECRET 和 API_MASTER_KEY（不能每次重启随机生成）
- 数据库密码不能用 123456
- HTTPS
- 自定义域名（可选）

## 四、推荐执行顺序

```
A（定时采集）→ B（AI内化）→ D（前台改造）→ C（多源扩展）→ E（个人网站）→ F（部署）
```

### 理由

1. **A 和 B 是核心价值** — 没有自动采集和 AI 内化，这个项目和普通 CMS 没区别
2. **D 紧跟着做** — 采集和内化做完后，前台要能展示这些新内容类型
3. **C 是扩展** — 先把论文这条线跑通，再加资讯源
4. **E 和 F 不急** — 个人网站是锦上添花，部署等功能稳定后再说

## 五、技术注意事项

### 数据库

- 现有 `ai_content` + `ai_content_external_ref` + `extra_json` 的三层结构基本够用
- 需要新增 `ai_fetch_log` 表记录采集日志
- 可能需要新增 `ai_paper_profile` 表存储论文特有字段（关联度评分、子方向分类等）

### 去重策略

- arXiv 论文：用 `arxiv_id` 去重
- HuggingFace 论文：用 `paper_id` 去重
- GitHub 仓库：用 `full_name` 去重
- RSS 文章：用 URL 去重
- 所有去重逻辑放在 Service 层，导入前先查询是否已存在

### AI 总结成本控制

- 每天新增 10-30 篇论文 × 每篇总结约 500-1000 tokens
- 可以设置"只对关联度评分 >= 6 的论文生成详细总结"
- 可以先用轻量模型做初筛，再用强模型做高分论文的详细总结
- MiMo API 相对便宜，适合批量任务

### 定时任务

- Spring Boot `@Scheduled` 足够初期使用
- 后期如果任务量大，可以迁移到 Quartz 或 XXL-Job
- 采集时间建议设在凌晨（国内凌晨 = 美国白天，新论文最多）

### 安全

- 现有的 JWT + AES-256-GCM 机制保持不变
- 定时采集不需要额外权限，后端内部调用即可
- 部署时必须配置固定的密钥环境变量
